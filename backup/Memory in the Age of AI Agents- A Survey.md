# Step 1：详细、通俗地讲解论文

## 1. 论文解决什么问题？

现在很多 Agent 论文都声称自己使用了“记忆”，但它们说的可能完全不是同一件事：

* 有的把历史对话放进向量数据库；
* 有的把成功经验总结成文字；
* 有的保存完整任务轨迹；
* 有的微调模型，让知识进入参数；
* 有的复用 KV Cache 或隐藏状态；
* 有的只是不断压缩当前上下文。

这些方法都被叫作“Memory”，但其存储位置、用途、更新方式和评测标准相差很大。

作者认为，应该问三个问题：

1. **记忆以什么形式存在？**
2. **记忆用来做什么？**
3. **记忆如何产生、检索和变化？**

这就是论文最核心的 **Forms–Functions–Dynamics** 框架。 ([arXiv][1])

## 2. 如何定义 Agent Memory？

论文把 Agent 看成一个不断与环境交互的系统。在时间步 $t$，Agent 根据：

* 当前观察 $o_t$；
* 当前任务 $Q$；
* 从记忆中取出的信息 $m_t$；

决定下一步动作：

$$
a_t=\pi(o_t,m_t,Q)
$$

记忆本身不是一个固定数据库，而是一个随时间变化的状态 $M_t$。

论文将记忆生命周期概括为三个操作：记忆形成、记忆演化、记忆检索

### 2.1 Memory Formation：记忆形成

$$
M_{\text{form}}^{t+1}=F(M_t,\phi_t)
$$

即：从当前对话、环境观察、工具调用结果或任务轨迹中，提取值得保存的信息。

### 2.2 Memory Evolution：记忆演化

$$
M_{t+1}=E(M_{\text{form}}^{t+1})
$$

新信息不能无脑追加到数据库中，还需要：

* 合并重复内容；
* 处理新旧记忆冲突；
* 删除过时信息；
* 将具体案例抽象成一般规律；
* 调整记忆结构；
* 降低低价值记忆的优先级。

一个好的记忆系统不应把信息当作绝对真理，而应更新版本、记录时间和适用范围。

### 2.3 Memory Retrieval：记忆检索

$$
m_t=R(M_t,o_t,Q)
$$

检索不是简单地做一次向量相似度搜索，而是包括：

* 根据当前任务构造查询；
* 选择应该搜索哪类记忆；
* 使用语义检索、关键词搜索或图遍历；
* 过滤无关或冲突记忆；
* 将记忆整理成适合 LLM 使用的上下文。

因此，Agent Memory 更像一个不断运行的操作系统，而不是一个只会读写的硬盘。 ([Moonlight][2])

## 3. Agent Memory 与相邻概念有什么区别？

### 3.1 Agent Memory 与 RAG

传统 RAG 通常是：

> 用户提出问题 → 从外部知识库搜索文档 → 生成答案。

知识库往往是预先构建的、相对静态的，主要目的是补充外部知识。

Agent Memory 更强调：

> Agent 在连续交互中自己积累、修改和使用内部经验。

当然，现在的 Agentic RAG 会自动决定何时搜索、搜索什么，因此二者边界正在变得模糊。 ([arXiv][1])

### 3.2 Agent Memory 与 Context Engineering

Context Engineering 关注：

> 在有限上下文窗口中，应该放入哪些内容，如何排序和压缩？

它主要解决当前推理的上下文组织问题。

Agent Memory 的范围更大，还包括：

* 跨任务持久化；
* 用户身份与偏好维护；
* 经验积累；
* 技能学习；
* 遗忘和记忆更新；
* 多 Agent 共享。

因此可以把 Context Engineering 看作 Agent 工作记忆的一种实现手段，而不能把它等同于完整的 Agent Memory。 ([Moonlight][2])

### 3.3 Agent Memory 与 LLM Memory

论文主要把纯模型内部的以下机制与 Agent Memory 区分开来：

* KV Cache 管理；
* 长上下文建模；
* Transformer 架构修改；
* 不具有持续演化机制的内部状态。

关键区别在于：Agent Memory 通常服务于一个持续行动的 Agent，并且会随着交互和任务经验不断变化。 ([Moonlight][2])

## 4. Forms——记忆以什么形式存在？

### 4.1 Token-level Memory：Token 级记忆

这是目前最常见、最容易工程实现的形式。

记忆被保存为显式、可读、可修改的对象，例如：

* 文本；
* JSON；
* 对话摘要；
* 用户画像；
* 任务轨迹；
* 知识图谱；
* 代码和技能；
* 图像或多模态 Token。

它最大的优点是：

* 可解释；
* 可人工检查；
* 容易增删改；
* 可以追踪记忆来源；
* 不需要修改基础模型。

缺点则是：

* 记忆不断增长；
* 检索成本逐渐增加；
* 可能检索到相似但无用的内容；
* 大量记忆进入上下文会带来 Token 成本；
* 记忆质量高度依赖提取和检索模块。

论文进一步按照结构将其划分为三类。 ([Moonlight][2])

#### 4.1.1 Flat Memory：平面记忆

每条记忆相对独立，例如向量数据库中的若干文本块：

```text
Memory 1：用户喜欢 Python。
Memory 2：上次安装 CUDA 时驱动版本不匹配。
Memory 3：这个项目使用 Conda 环境。
```

优点是简单、便于追加；缺点是随着记忆增多，很难表达复杂关系。

#### 4.1.2 Planar Memory：结构化记忆

使用一层结构连接记忆，例如：

* 图；
* 树；
* 表格；
* 时间线；
* 单层知识图谱。

例如：

```text
用户
 ├── 偏好：Python
 ├── 设备：ThinkStation
 └── 项目：Agent Memory
       ├── 使用模型：Qwen
       └── 当前问题：检索失败
```

它比单纯向量库更适合关系推理，但构建和维护成本更高。

#### 4.1.3 Hierarchical Memory：层次化记忆

记忆被分为多个抽象层级：

```text
原始轨迹
   ↓
任务摘要
   ↓
通用策略
   ↓
长期技能
```

例如，Agent 多次执行代码任务后：

* 底层保存完整操作轨迹；
* 中层保存每个任务的错误总结；
* 高层提炼出“部署前检查清单”。

这种结构更适合长期学习，但也最复杂：系统必须决定什么时候总结、总结到哪一层，以及如何避免抽象过程损失关键信息。

### 4.2 Parametric Memory：参数记忆

参数记忆是把知识或经验写入模型权重中。

#### 内部参数记忆

直接更新基础模型，例如：

* 持续预训练；
* SFT；
* 强化学习；
* 在线微调。

优点是推理时不一定需要额外检索；缺点是：

* 更新代价大；
* 难以解释；
* 很难删除某一条具体记忆；
* 可能发生灾难性遗忘；
* 高频更新不现实。

#### 外部参数记忆

不修改主模型，而使用：

* LoRA；
* Adapter；
* 小型辅助模型；
* 专门的记忆编码器。

例如可以给不同用户加载不同 LoRA，使个性化知识保存在独立模块中。

这种方法比修改整个模型更灵活，但仍然存在难以审计和精确删除的问题。 ([Moonlight][2])

### 4.3 Latent Memory：潜在表示记忆

这类记忆不是可读文本，而是：

* Hidden State；
* KV Cache；
* 连续向量；
* Memory Token；
* 压缩后的潜在表示。

论文将其概括为三种处理方式。 ([Moonlight][2])

#### Generate：生成新的潜在记忆

让一个编码器把长文本压缩成少量连续向量或特殊 Token。

优点是紧凑、节约上下文；缺点是压缩后不知道具体丢失了什么。

#### Reuse：复用已有状态

直接保存和复用过去的 KV Cache 或隐藏状态。

这样能保留较完整的内部计算结果，但缓存会快速增长，而且不同上下文之间未必能够直接兼容。

#### Transform：转换已有状态

对潜在状态进行：

* 剪枝；
* 聚合；
* 重编码；
* 选择重要 Token；
* 压缩 KV Cache。

本质是在信息完整性和计算效率之间寻找平衡。

### 我的看法

论文把“记忆到底存在哪里”说清楚了。

最实用的区别是：

| 形式          | 可读性 | 更新方便程度 |       推理成本 | 主要风险      |
| ----------- | --: | -----: | ---------: | --------- |
| Token-level |   高 |      高 | 检索和上下文成本较高 | 噪声、检索错误   |
| Parametric  |   低 |      低 |     使用时较自然 | 遗忘、不可精确删除 |
| Latent      |  很低 |     中低 |      可能更高效 | 信息损失、难以审计 |

未来真正强大的 Agent 很可能不是只使用一种，而是采用混合结构：

* Token 记忆负责可审计事实；
* 参数记忆负责稳定技能；
* Latent 记忆负责高效率工作状态。

## 5. Functions——Agent 为什么需要记忆？

### 5.1 Factual Memory：事实记忆

事实记忆回答：

> 过去发生过什么？世界和用户是什么样的？

它又可以包含：

* **情景记忆**：某次具体事件；
* **语义记忆**：从多个事件中形成的一般事实。

事实记忆主要保证 Agent 的一致性、个性化和跨会话连续性。 ([Moonlight][2])

### 5.2 Experiential Memory：经验记忆

经验记忆回答：

> 过去是怎么做的？什么方法有效？什么方法会失败？

论文将其进一步分成三类。

#### Case-based Memory：案例记忆

保存具体成功或失败案例，遇到相似任务时，可以直接复用整个案例。

#### Strategy-based Memory：策略记忆

不是保存完整轨迹，而是提炼策略。

这比具体案例更容易迁移到新任务。

#### Skill-based Memory：技能记忆

将经验进一步编译成可执行技能，例如：

* Python 函数；
* Shell 脚本；
* 工具调用流程；
* 工作流模板；
* API 使用方法。

此时 Agent 不只是“记得过去”，而是把经验转化成新的行动能力。 ([Moonlight][2])

### 5.3 Working Memory：工作记忆

工作记忆类似 Agent 当前使用的“草稿纸”，用于保存：

* 当前目标；
* 已完成步骤；
* 中间推理结果；
* 待验证假设；
* 当前计划；
* 工具返回结果；
* 尚未解决的子问题。

它的核心不是永久保存，而是在当前任务中保持目标和状态一致。 ([Moonlight][2])

---

### 我的看法

这是整篇论文里我认为最重要的分类。

很多系统所谓“增加记忆”，实际上只增加了事实记忆：

> 给 Agent 接一个向量数据库，让它能记住用户说过的话。

但 Agent 仍然会反复犯相同的操作错误，因为它没有：

* 经验记忆：总结过去为什么失败；
* 策略记忆：形成解决问题的原则；
* 技能记忆：把成功方法编译成工具。

所以，**记住事实不等于学会做事**。

真正的自进化 Agent，关键并不是记住更多历史，而是把历史转换成可复用的策略和技能。

## 6. Dynamics——记忆如何运行？

### 6.1 Formation：记忆形成

形成阶段决定“写什么”。

可能包括：

* **Perception**：把原始观察编码为可处理表示；
* **Extraction**：提取实体、关系、偏好和关键事件；
* **Summarization**：压缩长内容；
* **Abstraction**：从案例中提炼一般规律；
* **Reflection**：对执行过程进行反思。

最困难的问题是：

> 什么信息值得保存？

如果全部保存，记忆会爆炸；如果只保存摘要，可能丢掉关键细节。

### 6.2 Retrieval：记忆检索

检索阶段决定“读什么”。

通常包括：

1. 为记忆建立索引；
2. 根据任务构造查询；
3. 执行语义搜索、关键词搜索或图遍历；
4. 对候选结果重排；
5. 过滤冲突和低可信记忆；
6. 将检索结果整合进上下文。

这里有一个经常被忽视的问题：

> 检索内容与当前问题相似，不代表它对决策真正有用。

例如当前问题是“模型为什么输出 NaN”，向量检索可能找到大量包含“NaN”一词的记录，却未必找到真正相关的“学习率过大”经验。

因此，未来的记忆检索应该优化**决策价值**，而不只是文本相似度。

### 6.3 Evolution：记忆演化

演化阶段决定“如何修改已有记忆”。

包括：

* Consolidation：合并和巩固；
* Updating：更新事实；
* Reorganization：调整结构；
* Pruning/Forgetting：删除低价值内容；
* Generalization：提炼通用知识；
* Distillation：将大量案例压缩成策略或技能。

论文的重要观点是：

> 遗忘不是记忆系统的缺陷，而是记忆系统必须具备的能力。

没有遗忘机制的 Agent 最终会被过时、矛盾和低质量信息淹没。 ([Moonlight][2])

## 7. 用一个完整例子理解三维框架

假设有一个编程 Agent，连续帮助用户部署模型。

第一次部署失败：

```text
CUDA available: False
```

Agent 可以形成以下记忆：

### Forms

* Token-level：保存错误日志和解决方案；
* Parametric：通过后续训练学会 CUDA 排查流程；
* Latent：压缩当前任务中的大量终端输出。

### Functions

* Factual：用户使用 Ubuntu，GPU 驱动曾经损坏；
* Experiential：先检查 `nvidia-smi`，再检查 PyTorch CUDA 版本；
* Working：当前已修复驱动，下一步检查模型依赖。

### Dynamics

* Formation：从终端日志中提取失败原因；
* Retrieval：下次出现 CUDA 问题时找回该经验；
* Evolution：发现旧解决方法不适用于新驱动后，更新原有记忆。

这说明，一套完整 Agent Memory 不是一个向量数据库，而是一套包含**存储、提炼、搜索、更新和遗忘**的认知基础设施。

# Step 2：实验部分

## 1. 没有传统意义上的实验

这是一篇综述，而不是提出新模型的算法论文。

论文对应的“实验部分”，实际上是对以下内容的整理：

1. Agent Memory 基准；
2. 常用评测指标；
3. 开源 Memory 框架；
4. 不同工作之间评测协议不统一的问题。

作者本人也明确把“汇总基准与开源框架”作为论文贡献之一。 ([arXiv][1])

## 2. 事实记忆如何评测？

事实记忆通常使用长对话或多会话问答基准，例如综述讨论范围中的 LoCoMo、LongMemEval 等。

常见测试形式是：

> 在很早以前的对话中埋入一条信息，经过大量无关对话后，再询问 Agent。

例如：

```text
第 1 轮：我的狗叫 Coco。
……
第 300 轮：我的狗叫什么？
```

更困难的题目还会要求：

* 跨多个对话片段组合信息；
* 理解事件的先后顺序；
* 识别用户偏好的变化；
* 区分旧事实和新事实；
* 在大量干扰信息中找出关键记忆。

这类基准主要检验事实与情景记忆，但并不能充分说明 Agent 是否真正从过去的经验中学会了新的策略。 ([GitHub][3])

## 3. 经验记忆如何评测？

经验记忆应该关注：

> 使用过去经验后，未来任务是否做得更好？

例如，Agent 第一次完成网页任务时失败，将失败原因写入记忆；第二次遇到类似网站时，观察：

* 成功率是否提高；
* 工具调用次数是否减少；
* 是否避免重复错误；
* 是否能迁移到稍有不同的任务；
* 记忆是否产生负迁移。

但现有不少工作只展示“加入经验库后平均分提高”，没有真正区分提升来自：

* 更多上下文；
* 更好的提示词；
* 基础模型本身变强；
* 记忆机制确实学到了可迁移经验。

## 4. 工作记忆如何评测？

工作记忆通常关注长任务中：

* Agent 能否保持当前目标；
* 是否遗漏已完成步骤；
* 是否重复执行动作；
* 上下文压缩后是否保留关键信息；
* 使用多少 Token；
* 推理延迟和成本是否下降；
* 最终任务成功率是否受影响。

一个好的工作记忆系统不能只追求“压缩率”。

例如，把 100,000 Token 压成 1,000 Token 看起来非常高效，但如果恰好删除了任务约束，Agent 最终失败，那么这种压缩没有实际价值。

因此应联合衡量：

$$
\text{任务成功率}+\text{Token 成本}+\text{延迟}+\text{信息保真度}
$$

## 5. 开源框架整理

论文及其配套列表整理了大量代表性系统，例如：

* MemGPT；
* Mem0；
* Zep；
* A-MEM；
* MemoryBank；
* HippoRAG；
* 不同类型的图记忆、层次记忆和经验记忆系统。 ([GitHub][4])

这对工程人员很有用，可以根据三维框架选择系统：

| 需求        | 更关注的记忆能力                 |
| --------- | ------------------------ |
| 记住用户偏好    | Factual + Token-level    |
| 长期对话一致性   | Factual + Hierarchical   |
| 避免重复犯错    | Experiential + Strategy  |
| 自动积累工具能力  | Experiential + Skill     |
| 控制长上下文成本  | Working + Latent         |
| 保存可审计企业信息 | Token-level + Provenance |

# Step 3：简要汇报

这篇综述系统梳理了 AI Agent 的记忆研究，指出传统的长短期记忆分类已经不足，并提出形式、功能和动态三个分析维度。形式包括 Token、参数和潜在表示记忆；功能包括事实、经验和工作记忆；动态则涵盖记忆形成、检索、更新与遗忘。论文的核心价值是把记忆从简单数据库提升为 Agent 的基础认知能力，但目前分类边界仍有重叠，也缺少统一实验验证。未来值得研究的方向包括可学习的记忆管理、不同记忆形式的一致性，以及多 Agent 共享记忆的安全与可信问题。

该段概括基于论文的三维分类、评测资源和未来研究方向。 ([arXiv][1])
