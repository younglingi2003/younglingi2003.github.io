# 一、MALADE：几乎就是“一个主 Agent 一个影子”

MALADE 的结构是：

```text
DrugFinder ── Critic-DrugFinder
     ↓
DrugAgent  ── Critic-DrugAgent
     ↓
CategoryAgent ── Critic-CategoryAgent
```

每个主 Agent 生成：

* 回答；
* 推理步骤；
* 证据与理由。

对应 Critic 检查：

* 推理是否正确；
* 是否遵循指令；
* 证据是否充分；
* 输出格式是否合规。

如果不通过，主 Agent 根据反馈重新生成，直到 Critic 满意。作者明确将其描述为：**pairing each agent with a critic agent**。([arXiv][1])

所以，“给每个角色 Agent 配一个监督影子”这一层，已经高度撞车。

# 二、 Textual Equilibrium Propagation：更通用的“每节点一个影子”

这篇 ICLR 2026 工作比 MALADE 更接近一个通用方法。

它将 Multi-Agent 或 Compound AI 系统表示为图：

```text
Problem Analyzer → Code Generator → Test Generator → Final Validator
        ↑                ↑                ↑
    Local Critic     Local Critic     Local Critic
```

每个节点都有自己的本地 Critic，分别评价：

* 功能正确性；
* 约束满足；
* 上下文利用；
* 内部一致性；
* 推理透明度；
* 格式遵循。

各节点先在本地 Critic 的指导下反复改进，然后系统再通过全局目标协调这些局部结果。也就是说，它已经把“每个主 Agent 设置局部监督者”推广到了任意 Agent 工作流节点。([arXiv][2])

# 三、 ACC-Collab：专门训练一个敢于“抬杠”的影子

ACC-Collab 使用一个 Actor 和一个 Critic，并联合训练二者如何协作。作者发现，未经训练的 Critic 经常：

* 过于赞同 Actor；
* 不愿意明确指出错误；
* 只给空泛建议；
* 无法改变 Actor 的错误答案。

训练后的 Critic 更愿意提出实质性反对意见，并能显著提高 Actor 的正确率。([arXiv][4])

# 一、最直接的撞车：Sherlock

## Sherlock: Reliable and Efficient Agentic Workflow Execution

Sherlock 几乎明确提出了与您相同的三个问题：

* 工作流中哪些节点的错误最容易影响最终答案？
* 哪些节点值得付出额外成本进行验证？
* 对每个节点，应该动态选择哪一种验证器？

它把 Agent 工作流表示成图，节点是不同 LLM、工具、检索或推理操作，边表示信息依赖。论文强调，上游节点的局部错误会沿工作流传播并污染最终输出，因此不能简单地验证所有节点，也不能只验证最后一个节点。([arXiv][1])

它的完整结构实际上就是：

```text
局部节点输出
    ↓
判断该节点是否值得验证
    ↓
动态选择最合适的 Verifier
    ↓
验证或修正
    ↓
错误则回滚并重新执行下游节点
```

# 二、SupervisorAgent：覆盖“按需选择监督操作”

## Stop Wasting Your Tokens: Towards Efficient Runtime Multi-Agent Systems

SupervisorAgent 的结构也很接近您的设计：

```text
Agent-Agent 交互
Agent-Tool 交互
Agent-Memory 交互
        ↓
轻量级风险过滤器
        ↓
是否触发 Supervisor
        ↓
动态选择干预动作
```

它不会检查所有步骤，而是通过轻量过滤器识别关键交互位置，例如：

* 明确错误；
* Agent 陷入低效循环；
* 工具输出过长；
* 错误信息可能进入后续上下文。

只有触发这些风险条件时，Supervisor 才介入。([arXiv][2])

触发后，Supervisor 会根据：

* 全局任务；
* 当前 Agent 的局部任务；
* 最近执行轨迹；
* 当前输出；
* 全局轨迹；

动态选择动作：

$$
a\in
{
\text{approve},
\text{provide guidance},
\text{correct observation},
\text{run verification}
}
$$

其中：

* $`approve`$：不干预；
* $`provide_guidance`$：提供方向性提示；
* $`correct_observation`$：修正错误或压缩输入；
* $`run_verification`$：调用专门验证 Agent。

这已经是“对每个局部状态动态选择监督操作”，而且包含显式的 No-op。([arXiv][2])

# 三、From Spark to Fire：覆盖“传播风险驱动监督”

这篇论文直接研究：

> 一个局部错误如何沿着 LLM-MAS 的通信图传播，并最终形成系统级错误共识。

它将消息流表示成有向图，并分析：

* 错误的逐跳传播；
* 中心节点导致的放大；
* 下游上下文不断积累后的错误固化；
* 干预越晚，纠正代价越高。

论文发现，即使系统中存在 Reviewer 或 QA Agent，也不能保证阻止已经开始传播的错误。([arXiv][3])

## 它的防御机制

论文提出 Genealogy-Based Governance Layer：

```text
Agent 局部输出
      ↓
拆成原子 Claim
      ↓
构建 Claim 依赖与传播图
      ↓
风险与不确定性判断
      ↓
选择：
跳过验证 / 选择性验证 / 全面验证
      ↓
放行、阻断或回滚
```

它明确提出，在验证预算有限的情况下，应把昂贵验证分配给：

* 结构影响力高的 Agent；
* 汇总者、决策者等功能 Hub；
* 高不确定性 Claim；
* 下游传播风险较高的 Claim。([arXiv][3])

Balanced Policy 只验证来自高影响节点的可疑 Claim；普通节点的可疑信息则携带不确定性标签继续传播，而 Strict Policy 会验证所有可疑 Claim。([arXiv][3])
