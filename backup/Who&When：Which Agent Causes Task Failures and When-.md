# 论文解读：Which Agent Causes Task Failures and When?

> **论文发表信息**：ICML 2025，论文集 PMLR 267。官方代码仓库将其标注为 ICML 2025 Spotlight。([Proceedings of Machine Learning Research][1])

# Step 1：论文讲解

## 1. 论文想解决什么问题？

目前评估一个 LLM 多智能体系统，通常只能得到：

* 任务成功了；
* 任务失败了；
* 最终答案得分是多少。

但这对于调试系统并不够。

最后任务失败了，开发者真正关心的其实是：

1. **Who：到底是哪个 Agent 应该为失败负责？**
2. **When：它在哪一个步骤犯下了关键错误？**
3. 这个错误究竟是普通错误，还是**真正导致最终失败的决定性错误**？

过去通常需要开发者手动阅读几十甚至上百步运行日志。本文提出的核心任务，就是让另一个 LLM 自动完成这项调试工作，即：

> **Automated Failure Attribution：自动失败归因。**

论文不是简单判断“某一步是否有错误”，而是试图找到“哪个错误真正改变了最终结果”。([arXiv][2])

## 2. 什么是“决定性错误”？

论文首先区分了两个概念。

### 普通错误

某个 Agent 给出了不准确、不完整或不理想的输出。

但这个错误不一定会让整个任务失败，因为后面的 Agent 可能修正它。

### 决定性错误

如果把 Agent $i$ 在第 $t$ 步的错误动作替换成正确动作，并从该位置重新执行后续轨迹，系统就能从失败变为成功，那么这个错误就是决定性错误。

可以写成：

$$
\Delta_{i,t}(\tau)=1
\quad\Longleftrightarrow\quad
Z(\tau)=1; Z(\tau^{(i,t)})=0
$$

其中：

* $\tau$：原始失败轨迹；
* $Z(\tau)=1$：原轨迹失败；
* $\tau^{(i,t)}$：修正 Agent $i$ 在第 $t$ 步动作后的轨迹；
* $Z(\tau^{(i,t)})=0$：修正后任务成功。

如果轨迹中存在多个决定性错误，论文选择**时间最早的一个**作为主要失败原因。([arXiv][3])

### 一个简单例子

假设：

1. Planner 制定了正确计划；
2. WebSurfer 搜索到了错误年份；
3. Orchestrator 没有检查，直接采用；
4. Assistant 根据错误年份生成最终答案。

第 4 步的答案当然是错的，但它可能不是根因。

如果只修正第 4 步，Assistant 依然拿到错误资料，可能仍然无法回答正确；但如果修正第 2 步，后续流程可能全部恢复正常。因此论文会把：

> WebSurfer 在第 2 步的错误

视为决定性错误，而不是简单把责任归给输出最终答案的 Assistant。

### 我的看法

这是论文最有价值的概念之一。

它把“谁最后说错了”提升为“修复谁的哪一步才能挽救任务”，比普通错误检测更接近软件系统中的根因分析。

但是，论文后面实际上主要依靠人工阅读日志来判断这个反事实问题，并没有对每个样本真的执行“修正—重跑”实验。这使得它的形式化定义是因果性的，而实际标签更接近专家推断，二者之间存在一定差距。

## 3. Who&When 数据集

为了研究这个问题，作者构建了 **Who&When** 数据集。

数据集包含：

* 127 个 LLM 多智能体系统；
* 184 条失败任务轨迹；
* 每条轨迹标注责任 Agent；
* 标注决定性错误步骤；
* 提供自然语言失败原因；
* 任务来自 GAIA 和 AssistantBench。([arXiv][3])

### 数据构成

| 系统来源              |           任务来源 | 失败任务数 | 最大 Agent 数 |    日志长度 |
| ----------------- | -------------: | ----: | ---------: | ------: |
| CaptainAgent 自动生成 |           GAIA |    98 |          4 |  5–10 步 |
| CaptainAgent 自动生成 | AssistantBench |    28 |          4 |  6–10 步 |
| Magentic-One      |           GAIA |    30 |          5 | 5–130 步 |
| Magentic-One      | AssistantBench |    28 |          4 | 8–129 步 |

总计为 184 条失败轨迹。([arXiv][3])

这里需要注意一个容易被标题数字掩盖的细节：

> “127 个系统”并不等于127种独立的多智能体框架。

其中：

* 126 个是 CaptainAgent 针对126个任务分别自动生成的任务专用团队；
* 另外1个是人工设计的通用系统 Magentic-One；

CaptainAgent 会根据任务自动生成 Agent 名称、Prompt、工具和团队结构；这些自动生成系统及其 Agent 使用 GPT-4o。Magentic-One 则是一个相对成熟的人工设计系统，包含 Orchestrator、WebSurfer、FileSurfer 等角色。([arXiv][3])

### 标注过程

作者邀请了3位熟悉 AI Agent 的专家，采用三轮流程：

1. 每位专家独立标注责任 Agent、错误步骤和原因；
2. 对不确定样本进行讨论并达成共识；
3. 交叉检查其他专家的标注。

每位标注者花费约23.2至30.9小时；在初始标注中，约15%至30%的样本被标记为“不确定”，部分样本在讨论前仍存在明显分歧。

### 我的看法

这个标注成本本身就是论文的重要实验结果：

> 多智能体失败归因连人类专家都觉得困难，因此不能简单认为“让一个更强的 LLM 看完整日志”就能解决。

同时，15%至30%的不确定率也说明，这个任务可能天然存在多解：同一次失败中，多个 Agent 都可能负有不同程度的责任。

## 4. 三种自动失败归因方法

论文没有训练新的模型，而是设计了三种调用 LLM 的推理方式。

### 4.1 All-at-Once：一次性全局判断

把以下内容全部交给判断模型：

* 用户问题；
* 完整运行日志；
* 可选的正确答案；
* 所有 Agent 的行为。

然后要求模型直接输出：

* 责任 Agent；
* 决定性错误步骤；
* 判断理由。

优点：判断模型能够看到完整上下文，因此更容易比较不同 Agent 的行为，容易找到责任 Agent。

缺点：完整日志可能非常长，虽然“看到了”所有信息，很难精确找到某一个错误步骤。

### 4.2 Step-by-Step：逐步检查

模型从第一步开始逐条阅读日志。

每看到一步，就判断：

> 当前这一步是否已经发生决定性错误？

一旦回答“是”，算法立即停止，并返回当前 Agent 和步骤。

优点：模型每次只聚焦当前步骤，降低了在长上下文中寻找精确位置的难度，更容易定位错误步骤。

缺点：模型会过早做出决定。一个动作在当时可能有问题，但后续可能被其他 Agent 修复，容易把普通错误当成决定性错误，找错责任 Agent。

### 4.3 Binary Search：二分搜索

首先让模型判断错误位于日志的前半段还是后半段，然后不断缩小范围，直到剩下一步。

优点：在成本和上下文范围上处于前两者之间：
缺点：二分搜索默认给定半段日志，模型能够判断决定性错误是否位于其中，但多智能体错误往往需要结合“早期动作”和“后期后果”才能判断，早期是否错误，可能要到很后面才知道。

## 5. 三种方法为什么呈现完全不同的能力？

论文最重要的发现可以概括为：

| 任务            | 更需要什么信息            | 更合适的方法        |
| ------------- | ------------------ | ------------- |
| 判断哪个 Agent 负责 | Agent 之间的全局行为和因果关系 | All-at-Once   |
| 判断错误发生在哪一步    | 对局部步骤进行细粒度检查       | Step-by-Step  |
| 平衡成本和上下文      | 中等范围的信息            | Binary Search |

我的看法是，这是本文比具体准确率更值得重视的结论。后续方法不应继续要求一个 Prompt 同时输出责任 Agent 和错误步骤，而应该把问题显式分解为：

1. 全局责任候选识别；
2. 候选 Agent 内部的局部步骤搜索；
3. 反事实验证。

论文也尝试了类似的混合方案：先用 All-at-Once 找 Agent，再仅在该 Agent 的动作中使用 Step-by-Step 找步骤。([arXiv][3])

# Step 2：实验部分

## 1. 实验设置

作者设置了两种情况。

### 有正确答案

判断模型除了看到运行日志，还能看到任务的正确答案。

这对应系统开发阶段：开发者通常有验证集答案，可以分析系统为什么没有得到正确结果。

### 没有正确答案

判断模型只能看到问题和失败日志，无法参考标准答案。

这更接近系统上线后的自我反思：系统知道任务失败或结果不可靠，但不知道正确答案是什么。

主要实验使用 GPT-4o，同时测试：

* GPT-4 Turbo；
* GPT-4o-mini；
* Llama-3.1-8B/70B；
* Qwen2.5-7B/72B；
* OpenAI o1；
* DeepSeek R1。([arXiv][3])

评价指标包括：

* **Agent-Level Accuracy**：责任 Agent 是否预测正确；
* **Step-Level Accuracy**：精确步骤是否预测正确；
* **Step-Level Accuracy with Tolerance**：是否落在真实步骤附近。([arXiv][3])

## 2. 主实验结果

根据表1四种设置取平均，可以得到：

| 方法            |  Agent 准确率 |    精确步骤准确率 |
| ------------- | ---------: | ---------: |
| 随机选择          |     20.55% |     11.61% |
| All-at-Once   | **53.52%** |      8.70% |
| Step-by-Step  |     32.11% | **14.15%** |
| Binary Search |     40.54% |     13.59% |

这也解释了摘要中的两个数字：

* 最好的责任 Agent 准确率约为 **53.5%**，来自 All-at-Once；
* 最好的精确步骤准确率约为 **14.2%**，来自 Step-by-Step。

这两个最优数字实际上来自**不同方法**，说明当前没有一种方案可以同时解决 Who 和 When。([arXiv][2])

### 结果一：看完整日志更容易找到责任 Agent

All-at-Once 在四种条件下的 Agent 准确率为：

* 54.33%；
* 55.17%；
* 51.12%；
* 53.44%。

原因很直观：责任归属需要比较不同 Agent 的行为。如果只逐步看日志，模型可能发现某个局部错误就停止，而无法判断另一个 Agent 是否才是更根本的责任方。([arXiv][3])

### 结果二：看完整日志反而不容易找到精确步骤

All-at-Once 的平均步骤准确率只有8.70%，甚至低于随机基线的11.61%。

这说明把所有信息塞给模型，并不代表它能精确利用这些信息。完整日志中包含大量：

* 正确但无关的动作；
* 工具调用结果；
* 多个 Agent 的重复总结；
* 后续由早期错误引发的连锁错误。

模型容易找到“明显错误”，但不容易定位“最早的决定性错误”。作者将其与长上下文中的信息检索困难联系起来。([arXiv][3])

### 结果三：Step-by-Step 更适合精确定位，但仍然很差

Step-by-Step 在自动生成系统中的最好步骤准确率达到25.51%，但在长日志的 Magentic-One 系统中只有约7%至8.77%。

这意味着逐步检查在短日志上有一定效果，但面对几十至上百步的真实轨迹，依然远远达不到实用水平。

## 3. 日志越长，效果越差

作者把 Magentic-One 日志分为五个长度区间：

1. 5–17步；
2. 19–29步；
3. 31–49步；
4. 51–91步；
5. 93–130步。

随着日志变长，三种方法的 Agent 准确率和步骤准确率都持续下降，其中步骤准确率下降得尤其快。当日志达到93至130步时，三种方法的精确步骤准确率都接近0。([arXiv][3])

### 我的看法

这个结果非常重要。

它说明失败归因的瓶颈可能不是模型“不会推理”，而是模型无法在长轨迹中同时完成：

* 记住早期状态；
* 理解 Agent 之间的依赖；
* 区分原始错误和连锁错误；
* 判断后续是否存在恢复机会；
* 找到最早的不可逆转节点。

因此，单纯扩大上下文窗口未必有效，更需要轨迹压缩、结构化表示和动态检索。

## 4. 放宽步骤误差后会怎样？

作者允许预测步骤与人工标签相差若干步。

在 Magentic-One 上：

| 容忍范围 | All-at-Once | Step-by-Step | Binary Search |
| ---- | ----------: | -----------: | ------------: |
| ±1步  |      12.07% |   **14.66%** |        13.79% |
| ±2步  |  **19.83%** |       16.38% |        18.97% |
| ±3步  |  **30.17%** |       18.10% |        22.41% |
| ±5步  |  **43.10%** |       33.62% |        36.21% |

当要求非常精确时，Step-by-Step 略好；但容忍范围扩大后，All-at-Once 最好。

这说明 All-at-Once 往往能够找到：

> 错误大概发生在哪个区域。

但难以精确到具体哪一步。

## 5. 混合方法

作者先用 All-at-Once 找到责任 Agent，再只检查这个 Agent 执行的步骤。

在 Magentic-One 上：

| 方法            | Token 数 |  Agent 准确率 |      步骤准确率 |
| ------------- | ------: | ---------: | ---------: |
| All-at-Once   |  17,106 |     57.02% |      4.39% |
| Step-by-Step  |  87,720 |     35.96% |      7.90% |
| Binary Search |  34,659 |     43.97% |      6.90% |
| 混合方法          | 149,177 | **57.02%** | **12.28%** |

混合方法确实同时保留了较好的 Agent 准确率，并提高了步骤准确率，但 Token 成本最高，步骤准确率仍只有12.28%。([arXiv][3])

## 6. 更强的推理模型是否有效？

在 Magentic-One 数据上，作者比较了 GPT-4o、OpenAI o1 和 DeepSeek R1。

例如 All-at-Once：

| 模型          |  Agent 准确率 |  步骤准确率 |
| ----------- | ---------: | -----: |
| GPT-4o      |     54.31% |  4.39% |
| OpenAI o1   |     41.38% | 10.34% |
| DeepSeek R1 | **56.90%** |  3.45% |

Step-by-Step：

| 模型          |  Agent 准确率 |      步骤准确率 |
| ----------- | ---------: | ---------: |
| GPT-4o      |     33.62% |      7.90% |
| OpenAI o1   | **36.21%** | **13.79%** |
| DeepSeek R1 |     32.76% |      6.90% |

o1 在部分步骤定位任务上提升明显，但整体并没有稳定超越 GPT-4o；DeepSeek R1 也只在个别指标领先。

作者还发现，删除 Prompt 中要求模型显式分析和解释的推理指令后，性能会下降。例如自动生成系统中，All-at-Once 的 Agent 准确率下降7.4个百分点，Step-by-Step 的步骤准确率下降4.4个百分点。([arXiv][3])

### 我的看法

这个结果说明：

> 失败归因不是单纯依靠“模型更大”或“推理更长”就能解决的问题。

当前推理模型擅长在一个相对明确的问题中展开推导，但失败归因首先要求模型从长轨迹中建立正确的因果结构。如果输入表示本身混乱，增加推理长度可能只会产生更详细但未必正确的解释。

# Step 3：意见

## 总体优点

论文的主要贡献不在于提出了一个强方法，而在于：

1. 提出了一个重要但长期被忽视的问题；
2. 给出了较明确的任务形式化；
3. 构建并开放了带有 Who、When 和自然语言原因的失败轨迹数据；
4. 通过负面结果证明该问题远未解决。

作为问题定义与Benchmark论文，它有明显价值；作为方法论文，贡献则相对有限。

## 意见：形式上是因果归因，标签实际上只是专家的观察性判断

论文将决定性错误定义为：

> 修正该步骤并重新执行后，任务从失败变为成功。

这是一个严格的反事实定义。

但数据标注过程中，专家主要通过阅读已有失败日志来判断责任 Agent 和错误步骤，并没有对每个候选步骤真正执行干预、重跑后续轨迹。与此同时，标注者有15%至30%的样本不确定，也说明许多标签不是显然可验证的。([arXiv][3])

# Step 4：简要汇报

论文提出LLM多智能体自动失败归因任务，目标是定位“哪个Agent在第几步导致失败”。作者构建Who&When数据集，包含127个系统和184条标注轨迹。实验发现，全局判断更擅长找责任Agent，逐步判断更擅长找错误步骤，但最佳步骤准确率仅14.2%，强推理模型也未稳定提升。论文价值在于提出重要调试问题，但单一责任假设和数据多样性仍有限。
