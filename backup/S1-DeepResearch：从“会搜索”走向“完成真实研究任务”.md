# S1-DeepResearch：从“会搜索”走向“完成真实研究任务”

论文：**S1-DeepResearch: Beyond Search, Toward Real-World Long-Horizon Research Agents**

这是一篇发表于 **2026 年 6 月 13 日的 arXiv v1 预印本**。它没有提出新的模型架构，也没有设计新的强化学习算法；真正的贡献是一套面向 Deep Research Agent 的**数据构造、轨迹验证和监督微调方案**，以及基于该方案训练出的 S1-DeepResearch-32B。模型、15K 公开轨迹和推理代码已经发布。([arXiv][1])

我的整体判断是：

> 这篇论文抓住了一个很重要的问题：Deep Research 不应等同于“连续搜索很多次”。但论文最强的是系统工程与数据构造，最弱的是缺乏消融实验和训练细节，因此我们很难判断究竟是哪一部分真正产生了效果。

# Step 1：论文讲解

## 一、Deep Search 不等于 Deep Research

现在很多所谓 Deep Research Agent，训练任务实际上是：

1. 给出一个答案确定的问题；
2. 模型不断搜索网页；
3. 从网页中找到答案；
4. 输出一个短答案。

例如：

> 某位人物在某年担任某职位，他创办的机构叫什么？

这种任务确实需要多跳搜索，但最终目标仍然是**定位一个已经存在的事实**。作者把这种模式称为偏向 **Extractive Search，抽取式搜索**。

真正的 Deep Research 更像：

> 比较三种技术路线，限定最近五年的学术来源，分析成本、性能和风险，解释证据冲突，最后生成一份有引用、有结构、能用于决策的报告。

这种任务除了搜索，还要求：

* 理解复杂约束；
* 制定研究计划；
* 判断还缺少什么证据；
* 综合多个来源；
* 处理矛盾信息；
* 决定何时停止搜索；
* 使用文件、代码和专业工具；
* 最终生成报告、表格、PDF 或其他可交付物。

因此作者认为，当前 Agent 数据的主要缺口不是“搜索轨迹不够长”，而是没有覆盖完整的：

$$
\text{获取信息}
\rightarrow
\text{综合证据}
\rightarrow
\text{规划决策}
\rightarrow
\text{形成可交付结果}
$$

论文将 Deep Research 能力分为五类：

1. 长链复杂推理；
2. 深度研究指令遵循；
3. 深度研究报告写作；
4. 文件理解与生成；
5. 动态技能使用。([arXiv][2])

**我的看法：** 这个问题定义是合理的。现在很多 Deep Research benchmark 最终仍然是短答案 QA，只是在外面套了一层搜索工具。它们主要测“能不能找到”，而不充分测“找到之后能不能组织成有用的研究成果”。

## 二、核心方法

S1-DeepResearch 的训练流程可以概括成：

$$
\boxed{
\text{知识图谱生成任务}
\rightarrow
\text{Agent 执行任务}
\rightarrow
\text{轨迹改造}
\rightarrow
\text{多维验证}
\rightarrow
\text{SFT}
}
$$

具体分为三个阶段：

**阶段一：Graph-Grounded Task Formulation**

从知识图谱和网页中构造复杂但可验证的任务。

**阶段二：AgentLoop Rollout**

让一个 Agent 在真实工具环境中执行任务，得到完整的“思考—行动—观察—回答”轨迹。

**阶段三：Trajectory Verification**

检查答案、引用、约束、文件和技能使用是否正确，只保留通过验证的轨迹。

最后使用这些轨迹对 Qwen3-32B 做 SFT。([arXiv][2])

### 阶段一：用知识图谱“出题”

作者先从 Wikipedia 实体中选取种子实体，再利用 Wikidata 关系和开放网页进行扩展，建立一个包含实体、事件、属性和视觉概念的异构图。

例如，以“某种药物”为起点，图中可能逐渐出现：

* 药物；
* 发现者；
* 发现机构；
* 相关疾病；
* 临床试验；
* 监管事件；
* 竞争产品。

作者再通过社区发现，从大图中取出内部关系较密集的子图，用它作为一个复杂任务的知识骨架。([arXiv][2])

通俗来说，知识图谱在这里给 **出题系统** 使用。

它保证一个问题背后真的存在：

* 多个实体；
* 多条关系；
* 多步推理路径；
* 可以追溯的事实基础。

#### 论文生成两类任务 ：封闭式多跳 QA、开放式研究任务

**第一类：封闭式多跳 QA**

答案相对明确，可以判断对错。

例如：

> 某位曾在某机构任职的研究者，与另一位获得某奖项的科学家共同发表的论文，其发表年份是什么？

这种任务适合训练：

* 查询改写；
* 多跳检索；
* 候选验证；
* 精确答案生成。

**第二类：开放式研究任务**

答案不唯一，更接近真实用户需求。

例如：

> 比较三种医学影像分割方案，限定使用近五年论文，从精度、推理成本和临床可部署性三个角度分析，并以表格和建议报告的形式输出。

这种任务适合训练：

* 研究计划；
* 证据综合；
* 报告组织；
* 约束遵循；
* 决策输出。

作者的关键主张是：**封闭任务提供可验证性，开放任务提供真实性，两者必须结合。**([arXiv][2])

**给任务注入九类约束**

为了防止开放问题变成泛泛而谈，作者在生成问题前加入九种约束：

* 来源约束；
* 论证约束；
* 推理约束；
* 目标约束；
* 假设约束；
* 输出格式约束；
* 输出规模约束；
* 执行方式约束；
* 上下文约束。

例如，一个任务可能同时要求：

> 只能使用同行评议论文；
> 比较至少三种方法；
> 使用 SWOT 框架；
> 假设预算不超过某个值；
> 结果必须包含表格；
> 总长度不少于 2000 字；
> 必须调用代码计算；
> 面向非技术管理人员写作。

这样训练出来的 Agent 不只是学习“找到事实”，还学习在较长执行过程中持续记住用户约束。([arXiv][2])

#### 任务自进化：故意把问题变难

对于封闭式问答，作者进一步使用 Query Evolution 改写问题。

**实体语义改写**：把明显的人名、机构名，改成间接描述。

原问题：

> 爱因斯坦在哪一年获得诺贝尔奖？

改写后可能变成：

> 那位提出狭义相对论、曾在伯尔尼专利局工作的物理学家，在哪一年获得了相关国际科学奖项？

答案没变，但不能直接搜索原实体字符串。

**条件语义泛化** ：把精确条件变成更抽象的条件。

例如：

* “2012 年”变成“伦敦奥运会举办的年份”；
* “纽约”变成“美国人口最多的城市”；
* “37”变成“介于某两个范围之间的值”。

这样会迫使 Agent 先还原条件，再继续搜索。([arXiv][2])

#### 筛掉“不需要工具”的问题

生成复杂问题后，作者先关闭全部工具，让基础模型直接回答。

如果基础模型仅依赖参数知识就能回答，问题会被删除。作者称之为 **Parametric Knowledge Filtering**，目的是避免 Tool-use Bypass：模型绕过搜索工具，直接背出答案。([arXiv][2])

然后作者让 LLM 把预计推理过程转换成 Reasoning Graph，图中的节点包括：

* 实体；
* 约束；
* 中间结论；
* 最终答案。

作者根据四类图结构指标估计难度：

* 信息流复杂度；
* 反馈依赖复杂度；
* 推理宽度；
* 推理深度。

最后，每批任务只保留拓扑难度排名前 30% 的样本。([arXiv][2])

### 阶段二：让 Agent 真正执行任务

通过筛选的任务会进入 AgentLoop。Agent 可以使用九类工具：

* 网页搜索；
* 网页访问；
* 图片搜索；
* 学术搜索；
* 文件解析；
* 代码执行；
* Bash；
* 图片问答；
* 视频问答。

每一步都会形成：

$$
a_t=\text{模型的思考或工具动作}
$$

$$
o_t=\text{环境返回的搜索结果、网页、文件或执行结果}
$$

最终得到完整轨迹：

$$
(q,a_1,o_1,a_2,o_2,\ldots,a_T,y)
$$

其中 $q$ 是用户问题， $y$ 是最终回答。([arXiv][2])

论文的目标不是只保留最终答案，而是保留整个执行过程，用于教模型模仿“如何推进研究任务”。

#### 技能学习：让模型先读说明书再使用工具

论文还特别加入了 Skills 场景。

沙箱中会放置：

* 一个真正相关的 skill；
* 多个干扰 skill；
* 每个 skill 的 `SKILL.md` 使用说明。

Agent 不能仅凭工具名称猜用法，而需要：

1. 判断哪个 skill 与任务相关；
2. 打开对应 `SKILL.md`；
3. 阅读参数和执行流程；
4. 调用脚本；
5. 检查输出；
6. 根据结果继续处理。

这相当于训练模型学习：

> 面对一个没见过的专业工具，先阅读文档，再按文档执行。

它比把所有 API 用法预先写进系统提示词更接近真实 Agent 环境。([arXiv][2])

#### 对轨迹进行二次改造

原始 rollout 不一定直接适合作为训练数据，所以作者针对不同任务进行 refinement。

**报告任务**

先检查证据，再构建报告大纲和评价标准，最后重写报告，提高结构、引用和覆盖度。

**文件任务**

把纯文本回答改成真实文件任务，例如让 Agent 通过代码生成：

* PDF；
* DOCX；
* XLSX；
* HTML；
* SVG。

**多模态任务**

把原本依赖图片信息的文字问题，改造成真正带图片输入的问题。

**Skill 任务**

把问题中的代码、CSV 或配置内容改成附件，要求 Agent 自己发现和读取文件。([arXiv][2])

值得注意的是，报告 refinement 后，作者还会“刷新最后一个 Think，使其与最终报告保持一致”。([arXiv][2])

**我的看法：** 这可以让训练数据表面上更加连贯，但也存在风险：最终的思考可能是根据已经写好的答案反向补出来的，并不一定是实际产生答案的因果推理过程。这可能训练出漂亮的“事后解释”，而不是真实可靠的规划能力。

### 阶段三：多维轨迹验证

不同类型任务使用不同验证器。

#### 封闭式推理

检查：

* 最终答案是否正确；
* 中间推理是否完整；
* 是否存在逻辑错误。

#### 报告生成

检查：

* 正文引用与参考文献是否对应；
* 引用网页是否真的支持相关结论；
* 是否存在“引用看似正确但内容不支持”的情况。

#### 指令遵循

按九类约束逐项判断，只要违反任何一项，就过滤掉该轨迹。

#### 文件生成

检查：

* 是否真的调用代码；
* 执行是否成功；
* 是否生成目标文件；
* 文件内容是否满足语义要求。

#### Skill 使用

检查：

* 是否打开正确的 skill；
* 是否避开干扰 skill；
* 是否遵循其文档；
* skill 是否真正影响了最终结果。([arXiv][2])

论文大量使用 LLM-as-a-Judge，但也结合规则检查和执行结果。例如文件是否真实生成，可以通过文件系统确定，不完全依靠模型评分。([arXiv][2])

### 怎么训练？

基础模型是 **Qwen3-32B**。

训练方式很简单：只有 SFT。

训练时：

* 模型的思考、工具调用和最终回答参与损失；
* 搜索结果、网页内容和代码执行结果被视为环境输出，不计算损失。

直观上，相当于教模型：

> 在看到当前问题和已有观察后，模仿高质量轨迹中的下一步行动。

因此模型学习的不是某个固定答案，而是：

* 什么时候搜索；
* 搜索什么；
* 什么时候读文件；
* 什么时候执行代码；
* 什么时候停止；
* 如何形成最终报告。([arXiv][2])

### 数据规模需要澄清

论文的数据统计表分析了 **49,299 条轨迹**，其中包括：

| 数据类型     |    轨迹数 |
| -------- | -----: |
| 文本长链推理   | 18,246 |
| 多模态长链推理  |  5,203 |
| 报告生成     |  5,258 |
| 复杂指令遵循   |  7,877 |
| Skill 使用 |  4,208 |
| 文件理解与生成  |  8,507 |
| 总计       | 49,299 |

平均每条轨迹约有 9.7 次工具调用，平均总长度约 20,431 tokens。报告任务平均有 23.5 次工具调用，最终回答平均约 8,921 tokens。([arXiv][2])

但公开的数据集是 **S1-DeepResearch-15K**，包含 15,000 条中英文轨迹，总文件大小约 1.41 GB。([Hugging Face][3])

看起来，49,299 是作者内部构建或分析的完整数据池，15K 是公开子集；但论文没有清楚说明：

* 模型最终到底训练在 49,299 条还是 15K 条上；
* 15K 是如何从 49,299 条中抽样的；
* 各类别的训练比例；
* 是否还混入其他通用数据。

这是论文可复现性上的一个明显缺口。

# Step 2：实验部分

## 一、实验覆盖什么能力？

作者使用 20 个 benchmark，覆盖五大类能力。

### 1. 文本长链推理

* GAIA Text；
* BrowseComp；
* BrowseComp-ZH；
* xBench-DeepSearch；
* HLE Text。

### 2. 研究报告生成

* DeepResearchBench；
* DeepResearchBench II；
* ResearchRubrics。

### 3. 复杂指令遵循

* ComplexBench；
* 自建 DeepResearchIF。

### 4. 文件理解与生成

* GAIA File；
* GTA；
* 自建 FileSys。

### 5. 动态技能使用

* 自建 SkillUse。

另有六个多模态搜索和推理 benchmark。([arXiv][2])

为了尽量公平，作者为可统一运行的模型提供相同的九类工具。推理温度设为 0.85，top-p 为 0.95，重复惩罚为 1.1，每题最多允许 150 次工具调用，最大上下文长度为 128K。([arXiv][2])

## 二、文本长链推理：进步巨大

主要结果如下：

| 模型                      |     GAIA | BrowseComp | BrowseComp-ZH |   xBench |      HLE |       平均 |
| ----------------------- | -------: | ---------: | ------------: | -------: | -------: | -------: |
| Qwen3-32B               |     30.2 |        3.2 |           7.3 |     39.0 |      9.3 |     17.8 |
| Qwen3-235B              |     33.0 |       11.5 |          13.5 |     45.0 |     11.8 |     23.0 |
| **S1-DeepResearch-32B** | **72.8** |   **36.7** |      **48.4** | **79.3** | **30.3** | **53.5** |
| GPT-5.2                 |     68.0 |       65.8 |          76.1 |     83.0 |     45.5 |     67.7 |
| Gemini-3.1-Pro          |     70.9 |       85.9 |          63.3 |     80.0 |     51.4 |     70.3 |

S1 相比 Qwen3-32B 的平均成绩从 17.8 提升到 53.5，甚至明显超过 Qwen3-235B。说明 32B 模型经过 Agent 轨迹 SFT 后，工具调用和多跳搜索行为确实发生了根本变化。([arXiv][2])

更准确的结论是：

> S1 在部分任务上达到闭源模型附近，尤其是 GAIA 和 xBench；但在最困难的开放网页检索任务上仍有明显差距。

作者本人也承认 BrowseComp 和 HLE 上的差距，并认为后续可能需要强化学习。([arXiv][2])

## 三、报告生成：进步大

| 模型                      | DeepResearchBench |   DRB II | ResearchRubrics |       平均 |
| ----------------------- | ----------------: | -------: | --------------: | -------: |
| Qwen3-32B               |              36.0 |     27.8 |            41.7 |     35.2 |
| Qwen3-235B              |              38.4 |     29.7 |            43.9 |     37.3 |
| **S1-DeepResearch-32B** |          **46.5** | **41.7** |        **58.7** | **48.7** |
| GPT-5.2                 |              49.4 |     45.2 |            59.7 |     51.4 |
| Claude-4.6              |              46.7 |     51.9 |            60.5 |     53.0 |

S1 从基座的 35.2 提升到 48.7，与一些几百 B 参数的开源模型接近，也接近部分闭源模型。([arXiv][2])

这组实验比较支持作者的核心观点：

> 如果训练数据包含完整的证据收集、报告规划和最终写作过程，小模型也可以显著改善长报告能力。

不过报告指标大多依赖自动评估或 LLM judge，48.7 和 51.4 的差距是否对真实用户有感知，还需要人类专家评测。

## 四、复杂指令遵循：提升非常明显

在自建 DeepResearchIF 上：

| 模型                      | 严格 Query-Level Accuracy | Constraint-Level Macro Avg |
| ----------------------- | ----------------------: | -------------------------: |
| Qwen3-32B               |                     4.1 |                       28.9 |
| Qwen3-235B              |                     7.4 |                       54.9 |
| **S1-DeepResearch-32B** |                **25.2** |                   **74.3** |
| GPT-5.2                 |                    56.4 |                       89.2 |

严格准确率要求一个样本里的所有约束全部满足，只违反一条就算失败，因此分数普遍较低。S1 从 4.1 提升到 25.2，说明九维约束注入数据确实可能增强长程约束保持。([arXiv][2])

但这里也最需要警惕“训练—测试同构”：

* 训练数据就是按九类约束生成；
* 自建 benchmark 也按相似的九类约束评价。

模型可能学会了作者定义的约束模式，而不一定泛化到完全不同的真实用户指令。

## 五、文件任务：属于比较实用的提升

| 模型                      | GAIA File |      GTA |  FileSys |       平均 |
| ----------------------- | --------: | -------: | -------: | -------: |
| Qwen3-32B               |      24.2 |     70.3 |     44.7 |     46.4 |
| **S1-DeepResearch-32B** |  **62.9** | **68.0** | **69.3** | **66.7** |
| GPT-5.2                 |      62.9 |     73.8 |     48.7 |     61.8 |
| Claude-4.6              |      79.0 |     77.9 |     56.2 |     71.0 |

S1 在 FileSys 上达到 69.3，明显高于基座和一些闭源模型。([arXiv][2])

但 FileSys 的 FileAns 主要评估内容语义，不强调：

* 排版；
* CSS；
* 页面美观；
* 图表视觉质量；
* 专业文档格式。

也就是说，一个 PDF 只要内容正确，即使排版很差，仍可能获得较高评价。([arXiv][2])

所以论文证明的是：

> S1 更会成功生成“内容基本正确的文件”。

还不能直接推出：

> S1 能生成专业级、高质量、可直接交付的文件。

## 六、Skill 使用：很有研究价值，但 benchmark 是自建的

| 模型                      |      有附件 |      无附件 |       平均 |
| ----------------------- | -------: | -------: | -------: |
| Qwen3-32B               |     45.6 |     44.3 |     44.9 |
| **S1-DeepResearch-32B** | **69.7** | **71.7** | **70.1** |
| DeepSeek-V3.2           |     70.1 |     71.6 |     70.8 |
| GPT-5.2                 |     55.6 |     69.7 |     62.7 |

32B 的 S1 接近 DeepSeek-V3.2，并超过 GPT-5.2 的表中平均分。([arXiv][2])

SkillUse 不只评最终答案，还看：

* 是否选中正确 skill；
* 是否阅读说明；
* 是否避开干扰 skill；
* 是否遵循流程；
* 是否有效利用中间结果。

这个评价思想很好，因为它试图回答：

> 模型成功，是因为真正使用了外部技能，还是碰巧靠自身知识回答了？

不过，由于 SkillUse 是作者自建且尚缺乏广泛外部验证，这项结果更适合看作“有希望的内部证据”，而不是确定的行业领先结论。

## 七、多模态结果：能做，但不强

S1 本身是文本 LLM，不原生输入图片，而是先调用图片理解工具，把图片转换成文字，再交给语言模型。

这种方案的优点是不用重新训练多模态骨干；缺点是视觉信息会经过一次有损压缩。

例如 S1 的结果包括：

* LiveVQA：67.7；
* MM-Search：54.4；
* BrowseComp-VL：39.1；
* MM-BrowseComp：19.2；
* HLE-VL：15.2。

整体明显低于部分原生多模态模型和专门训练的多模态搜索 Agent。作者也承认，工具生成的文本描述难以保留细粒度视觉定位、空间关系和跨模态对齐信息。([arXiv][2])

**我的看法：** 这部分更像“证明统一 Agent 工作流可以临时接入视觉工具”，而不是证明模型已经获得真正的多模态研究能力。

## 八、测试时扩展

作者比较了 Pass@1 与 Pass@3。多采样几条轨迹后，五个文本长链推理任务均有所提升。

原因很直观：

* 第一次可能选错搜索入口；
* 第二次可能使用不同关键词；
* 某次遗漏的证据，另一次可能找到；
* 多条轨迹增加了覆盖正确路径的概率。([arXiv][2])

但这个实验比较弱，因为没有回答：

* Pass@3 的额外 token 成本是多少；
* 工具调用次数增加多少；
* 相同预算下，三次短轨迹是否优于一次长轨迹；
* 如何从三个答案中选择最终结果；
* 提升是否有统计显著性。

所以它只能说明“多采样有帮助”，还不能说明 S1 具有特别强的 test-time scaling 能力。

# Step 3：优点、评价

## 优点一：问题定义比单纯 Deep Search 更接近真实需求

论文明确把以下能力纳入统一框架：

* 搜索；
* 证据综合；
* 长程约束保持；
* 报告生成；
* 文件生成；
* 外部 skill 使用。

它没有把 Deep Research 简化成“多调用几次搜索 API”。这是一个重要而正确的研究方向。([arXiv][2])

## 优点二：数据构造流程完整且有系统性

从知识图谱出题、任务进化、难度筛选，到工具 rollout、场景 refinement 和多维验证，流程相对完整。特别是引用支持检查、文件执行检查和 skill 使用检查，比只判断最终答案更合理。([arXiv][2])

## 优点三：仅用 SFT 就让 32B 模型产生大幅行为变化

S1 没有依赖复杂 RL，而是在多个维度显著超过 Qwen3-32B，部分任务甚至超过远大于自身的模型。这至少说明：

> 对 Agent 而言，高质量、完整的行为轨迹可能比单纯扩大参数量更重要。([arXiv][2])

## 优点四：模型、数据和推理框架已公开

公开的 15K 数据包含完整消息轨迹，支持中英文；官方仓库也提供推理和工具环境部署方式。这比只公布 API demo 的工作更有价值。([Hugging Face][3])

## 评价

这篇论文提出了一个值得重视的观点：

> Deep Research Agent 的训练目标不应只是“找到答案”，而应覆盖从理解需求、制定计划、收集证据，到生成报告、文件和可执行成果的完整工作流。

它用相对系统的数据构造方法和纯 SFT，使 Qwen3-32B 在多个 Agent 任务上产生了非常明显的提升。这说明高质量 Agent 轨迹数据确实可能大幅改善中等规模模型。

但目前最核心的问题也很明确：

1. **49K 与公开 15K 的关系、训练配方和生成模型不透明；**
2. **关键优势集中于与训练体系高度同构的自建 benchmark；**
3. **大量依赖 LLM judge，缺少人工一致性、方差和成本分析；**

因此，我会把它看作一篇**很有价值的 Agent 数据与系统工作**，而不是已经充分验证的通用 Deep Research 学习理论。它最适合启发后续研究的地方，是把研究重点从“增加搜索长度”，转向：

> **如何构造覆盖完整工作流的轨迹，以及如何让模型在证据获取、约束遵循、停止决策和成果交付之间进行有效协调。**
