LLM多智能体通信：从“多个Agent聊天”走向“构建真正协作的智能体系统”

前言

标题： Beyond Self-Talk: A Communication-Centric Survey of LLM-Based Multi-Agent Systems

标签： 多智能体；智能体通信；通信协议；多智能体协作

摘要： 这篇论文最本质的思想是：理解一个多智能体系统，不能只看里面有多少个 Agent、每个 Agent 扮演什么角色，更应该看它们为什么通信、与谁通信、何时通信、以什么形式通信，以及通信内容如何影响整个系统。 作者据此提出一个“以通信为中心”的 LLM 多智能体分析框架，将通信分为系统级和系统内部两个层次：系统级包括通信架构、通信目标和通信协议，系统内部包括通信策略、通信范式、通信对象和通信内容。论文进一步梳理了 MCP、A2A、ANP 等协议，并讨论通信效率、扩展性、安全性、多模态和评测等问题。当前 arXiv 版本为 2026 年 5 月修订的 v3，论文已被 Frontiers of Computer Science 接收。(arXiv)

Step 1：论文解读

1：想解决什么问题

1.1：过去的综述“看系统”，但没有真正“看通信”

过去的 LLM 多智能体综述通常采用两类整理方式：

第一类按照系统结构整理，例如 Agent 的角色、记忆、规划、工具调用、工作流和应用场景；第二类按照领域整理，例如软件开发、社会模拟、推荐系统和科学研究。

这些整理方式当然有价值，但它们容易忽略一个核心问题：

多个 Agent 为什么能够形成一个“系统”，而不是几个彼此独立的大模型调用？

答案不是“因为 Agent 数量多”，而是因为它们之间存在某种信息交换和行为协调机制。通信决定了任务如何分解、信息如何传播、意见如何冲突、错误如何扩散，以及最终由谁做出决策。作者因此主张将通信视为 LLM-MAS 的第一等研究对象，而不是工作流图中的几条箭头。(arXiv)

我的看法： 这个切入点是合理且有价值的。很多所谓的多智能体系统，本质上只是“多个 Prompt 串联”；如果不分析通信机制，就很难解释增加 Agent 后为什么有时提升、有时反而变差。

1.2：提出两层、七维框架

作者将 LLM 多智能体通信分成两个层次：

层次 维度 要回答的问题
系统级通信 通信架构 Agent 被怎样组织，谁可以与谁通信？
系统级通信 通信目标 Agent 为什么需要通信？
系统级通信 通信协议 消息依据什么标准流动？
系统内部通信 通信策略 Agent 在什么时候、按照什么顺序通信？
系统内部通信 通信范式 消息怎样表示、传输和解释？
系统内部通信 通信对象 Agent 与谁或什么交互？
系统内部通信 通信内容 Agent 具体交换什么信息？

可以将它记成一句话：

系统级决定“组织方式、通信目的和基础规则”，系统内部决定“何时说、怎么说、对谁说、说什么”。

这是全文最重要的框架，也是这篇综述区别于一般 LLM-MAS 综述的核心贡献。(arXiv)

2：从单智能体到多智能体，通信为什么越来越重要

2.1：单个 Agent 的基本结构

论文将单个 LLM Agent 概括为三个主要部分：

  • Brain：以 LLM 为核心，负责推理、规划，并结合短期记忆和长期记忆；
  • Perception：将文本、图像、音频或环境状态转换为模型可处理的信息；
  • Action：输出文本、调用工具、访问 API，或者控制外部设备。

单 Agent 的问题在于：它的知识、上下文、推理和验证过程都集中在一个模型内部。当任务复杂、上下文很长，或者需要不同专业能力时，单 Agent 很容易出现规划失败、错误无法发现和上下文超限等问题。(arXiv)

2.2：传统 MAS 给了“协调骨架”，LLM 给了“自然语言接口”

传统多智能体系统已经研究了很长时间，典型方法包括：

  • 基于信念、愿望、意图的 BDI Agent；
  • Contract Net、拍卖和协商协议；
  • 多智能体强化学习中的集中训练、分散执行；
  • 图神经网络中的隐向量通信。

这些方法的优点是结构清楚、规则明确，但通信内容往往是任务专用的符号、动作或低维向量，难以适应开放环境。

LLM-MAS 的变化在于：Agent 可以直接使用自然语言表达计划、理由、批评、承诺和不确定性。因此，自然语言不仅是模型的输入输出，也逐渐成为多智能体系统中的一种通用协调媒介。(arXiv)

3：系统级通信之一——通信架构

通信架构决定信息能够沿着什么路径流动。论文总结了五类架构。(arXiv)

3.1：扁平架构

所有 Agent 基本平等，可以点对点交流，没有固定的管理者。

例如多个 Agent 分别提出答案、相互批评，再形成共识。

优点：

  • 灵活；
  • 没有单一控制瓶颈；
  • 容易产生不同观点。

缺点：

  • Agent 数量增加后，通信边数迅速增加；
  • 容易出现重复讨论；
  • 缺少明确的最终决策者。

扁平结构适合规模较小、需要头脑风暴或相互验证的系统。

3.2：层级架构

高层 Agent 负责规划、分配和监督，低层 Agent 负责具体执行。

例如软件开发系统中，管理 Agent 分配需求分析、编码和测试任务，最终再汇总结果。

优点：

  • 分工明确；
  • 适合复杂长流程；
  • 便于控制权限和责任。

缺点：

  • 上层 Agent 可能成为瓶颈；
  • 上层判断错误会影响整个系统；
  • 信息在逐层汇总时可能失真。

3.3：团队架构

Agent 被分成若干专业团队，每个团队负责一个子领域，例如开发组、测试组和评审组。

它与简单层级结构的区别在于，团队内部可能是平等协作的，而团队之间存在任务边界。

这种结构适合需要多种专业能力的任务，但跨团队信息同步会产生额外成本。

3.4：社会架构

社会架构不只强调完成一个明确任务，而是让大量 Agent 在规则、资源和社会关系中持续互动。

例如 Agent 可以形成朋友关系、联盟、市场交易或政治协商，并由微观互动产生宏观现象。

它适合社会模拟和涌现行为研究，但系统行为难以预测和控制。

3.5：混合架构

混合架构结合多种结构，例如：

  • 总体上采用层级管理；
  • 局部任务由平等 Agent 协作；
  • 必要时动态增加监督者；
  • 某些 Agent 可以临时加入或退出。

论文认为混合架构可能是未来的重要方向，因为现实任务往往无法被纯扁平或纯层级结构完整覆盖。(arXiv)

4：系统级通信之二——通信目标与通信协议

4.1：通信目标：合作、竞争和混合

4.1.1：合作

所有 Agent 具有共同目标。

合作又包括两种形式：

  • 直接合作：Agent 共享信息、整合结果，尽快完成共同任务；
  • 通过辩论合作：Agent 虽然目标一致，但会主动质疑其他 Agent 的推理。

这里的“辩论”并不等于竞争。辩论 Agent 最终仍然希望得到同一个正确答案，只是通过制造认知冲突来降低错误率。

4.1.2：竞争

Agent 具有冲突目标，或者争夺资源、收益和影响力。

竞争可能促使 Agent 形成更复杂的策略，但也会带来欺骗、攻击、串谋和不稳定等问题。

4.1.3：混合目标

现实环境中经常同时存在合作与竞争。例如多个公司共同制定行业标准，但又希望自己的方案占据主导地位；多个 Agent 可以暂时结盟，也可以随着资源变化而重新竞争。

混合目标比纯合作更接近真实社会，但对机制设计、安全监控和收益分配提出了更高要求。

锐评： 当前 LLM-MAS 研究明显偏向“大家一起完成任务”，对真正存在利益冲突的 Agent 研究不足。纯合作环境中的成功，不代表系统在开放环境中仍然可靠。

4.2：通信协议：MCP、A2A 与 ANP

论文介绍了三个新兴协议。(arXiv)

协议 主要连接对象 核心作用 典型特点
MCP Agent 与工具、数据源 访问外部上下文和能力 JSON-RPC、客户端—服务器
A2A Agent 与 Agent 能力发现、任务委托和结果返回 Agent Card、异步任务
ANP 开放网络中的异构 Agent 去中心化发现、身份验证和通信 DID、JSON-LD、端到端加密

MCP

MCP 更接近“Agent 如何标准化地使用外部工具”。Agent 不需要针对每个数据库、文件系统或 API 编写完全不同的接入逻辑，而是通过统一接口请求资源。

因此，MCP 解决的主要是：

Agent 怎样与工具和上下文通信？

A2A

A2A 更关注 Agent 之间的任务协作。Agent 可以公开自己的 Agent Card，说明自己具有什么能力；其他 Agent 发现它以后，可以向其委派任务，并接收状态更新和结果。

它主要解决：

哪个 Agent 能完成这个任务，以及怎样把任务交给它？

ANP

ANP 面向更加开放、去中心化的 Agent 网络。它使用去中心化身份进行身份发现和验证，并使用结构化语义表示交换信息。

它希望解决：

在没有统一中心管理者的开放网络中，陌生 Agent 如何发现、验证并安全通信？

我的看法： 论文将三者统一放在“多智能体通信协议”下便于理解，但三者实际上处于不同层次。MCP 偏工具访问，A2A 偏任务委托，ANP 偏身份和网络互联，它们未必是互相替代的竞争协议，更可能在同一个系统中组合使用。

5：系统内部通信之一——Agent 应该按照什么顺序交流

论文总结了三种主要通信策略。(arXiv)

5.1：逐个通信

Agent 按顺序工作，后一个 Agent 可以看到前面 Agent 的结果。

例如：

$$ A_1 \rightarrow A_2 \rightarrow A_3 \rightarrow A_4 $$

优点是上下文清晰，每一步都可以继承之前的信息；缺点是延迟随步骤增加，而且前面产生的错误可能被后续 Agent 不断继承和放大。

论文明确指出，逐个通信存在累积错误传播和语义漂移风险。(arXiv)

这与你之前关注的“决定性错误影响后续步骤”非常相关：一个错误是否具有决定性，往往不仅取决于该步骤本身，还取决于它是否改变了后续 Agent 的状态、假设和搜索空间。

5.2：并行通信

多个 Agent 同时处理任务，然后再汇总结果。

例如：

$$ {A_1,A_2,A_3,A_4}\rightarrow \text{候选结果集合} $$

优点是速度快、观点多样，而且某个 Agent 的错误不会立即污染其他 Agent。

主要问题有两个：

  • 状态过时：Agent 推理时看不到其他 Agent 的最新结果；
  • 结果冲突：多个输出重复、矛盾，需要额外的裁决机制。

5.3：并行通信加总结者

多个 Agent 先并行工作，再由一个 Summarizer 进行压缩、对齐和决策。

这种结构试图兼顾并行效率和全局一致性，但总结者会成为新的中心节点：

  • 总结者可能遗漏少数 Agent 的正确意见；
  • 总结者可能错误理解原始信息；
  • 总结者一旦幻觉，错误会被当成“全局共识”广播。

锐评： Summarizer 并没有真正解决通信问题，而是把“多 Agent 的冲突”压缩成了“一个总结 Agent 是否可靠”的问题。系统看似形成共识，实际上可能只是中心节点覆盖了不同意见。

6：系统内部通信之二——通信范式、对象和内容

6.1：通信范式

论文区分三种通信范式。(arXiv)

消息传递

Agent 直接向另一个 Agent 或一组 Agent 发送消息。

它简单直接，但 Agent 数量较多时容易出现消息过载、重复传播和状态不同步。

言语行为

消息不仅传递事实，还执行某种行为，例如：

  • 请求；
  • 承诺；
  • 拒绝；
  • 建议；
  • 质疑；
  • 指令。

例如“我认为答案是 A”只是传递观点，而“请你验证答案 A”则同时改变了接收者的任务状态。

言语行为适合协商和动态任务分配，但自然语言中的意图可能模糊。接收者一旦错误理解“建议”“命令”或“承诺”，便可能引起死锁或级联错误。

黑板机制

所有 Agent 通过一个共享空间读写信息。

它类似一个公共数据库、任务面板或共享记忆。Agent 不必逐一发送消息，只需将结果写入黑板，其他 Agent 按需读取。

黑板可以减少点对点消息数量，但需要处理权限、版本冲突、错误信息污染和中心化瓶颈。

我的看法： 这里的分类并不完全正交。“消息传递”和“黑板”属于信息传输机制，而“言语行为”属于消息语义。一个 Agent 完全可以通过黑板发布一条“请求”类型的言语行为，因此三者并不是严格互斥的同级类别。

6.2：通信对象

论文将通信对象分成四类：(arXiv)

对象 含义
自身 自我反思、内部规划、检查自己的推理
其他 Agent 信息交换、协商、监督和任务合作
环境 读取传感器、工具结果、模拟器状态和外部反馈
人类 接收人类指令、反馈、审批和纠正

这个划分提醒我们，多智能体通信不只是 Agent 互相聊天。Agent 对环境的观察、人类对 Agent 的反馈，以及 Agent 自己的反思，都可能改变后续协作过程。

6.3：通信内容

论文将通信内容分成显式和隐式两类。(arXiv)

显式通信

包括：

  • 自然语言；
  • 代码；
  • JSON 等结构化数据；
  • 明确的计划、结果和任务指令。

自然语言灵活，但容易歧义；结构化数据更准确，却要求所有 Agent 遵守统一格式。

隐式通信

Agent 不直接说出信息，而是让其他 Agent 从行为或环境变化中推断。

例如：

  • 一个 Agent 改变策略，暗示它不再相信原方案；
  • 多数 Agent 选择某个行动，形成隐式投票；
  • 工具返回错误，促使其他 Agent 调整任务；
  • 市场价格、交通状况或模拟环境状态影响 Agent 决策。

隐式通信能够降低消息开销，也适合竞争环境中保留私有信息，但接收者可能过度解读或错误理解这些信号。

我的看法： 显式与隐式通信的区分很有研究价值。现在大多数 LLM-MAS 只记录文本消息，但真正的错误传播可能同时发生在行为、工具状态、共享内存和环境反馈中。

7：作者认为未来最需要解决什么

论文提出六类主要方向。(arXiv)

7.1：优化系统结构

未来系统需要根据任务动态决定：

  • 使用多少 Agent；
  • 哪些 Agent 可以通信；
  • 是否增加监督者；
  • 什么时候并行；
  • 什么时候集中总结；
  • 哪些消息值得继续传播。

固定拓扑很难同时兼顾性能、延迟和成本。

7.2：研究合作与竞争的平衡

过度合作可能造成群体迎合，所有 Agent 迅速接受第一个答案；过度竞争又可能造成欺骗和资源浪费。

真正有价值的问题不是“合作好还是竞争好”，而是：

在什么状态下应该引入竞争，以及竞争到什么程度后应重新转为合作？

7.3：统一或兼容通信协议

随着 MCP、A2A、ANP、ACP 等协议出现，不同 Agent 系统可能形成新的协议孤岛。

论文希望未来形成类似互联网协议栈的标准化层次，而不是让每个框架都定义自己的消息格式。

7.4：多模态通信

未来 Agent 需要交换的不只是文字，还包括图像、视频、语音、轨迹和传感器状态。

难点不只是“模型能否看图”，而是不同 Agent 能否对同一多模态证据形成一致理解。

7.5：通信安全

单 Agent 安全主要关注输入输出；多 Agent 安全还要考虑消息伪造、中间人攻击、身份冒充、恶意 Agent 和错误信息级联传播。

一条被篡改的消息可能通过通信网络扩散，污染整个系统。论文因此主张在通信层加入身份认证、端到端加密、权限控制和来源追踪。(arXiv)

7.6：新的评测体系

现有评测通常只看最终任务是否完成，但无法回答：

  • Agent 之间传输了多少冗余信息；
  • 哪个 Agent 首先产生错误；
  • 错误传播了多远；
  • 系统能否抵抗过期或恶意消息；
  • 多 Agent 是否真正产生了协同增益。

作者认为评测应该同时覆盖个体能力、通信效率、集体收益、鲁棒性、公平性和安全性。(arXiv)

3：架构与目标表揭示了什么

Table 1 将论文放入五种架构和三种目标的组合中。

这支持了作者关于“Agent 竞争仍然研究不足”的判断。(arXiv)

4：协议对比表说明了什么

Table 2 从拓扑、发现机制、消息格式、安全性、优点、限制和应用场景等角度比较 MCP、A2A 和 ANP。(arXiv)

它最有价值的结论不是“哪个协议最好”，而是三个协议服务于不同层次：

  • MCP 适合工具和数据访问；
  • A2A 适合企业内部任务委托；
  • ANP 适合开放网络中的去中心化发现与互联。

因此，实际系统更可能形成分层协议栈，而不是三选一。

5：内部通信分类表揭示了什么

从 Table 3 的标注可以观察到几个总体趋势：(arXiv)

第一，消息传递是最常见的通信范式。多数系统本质上仍然依赖 Agent 之间发送文本或结构化消息。

第二，显式通信远多于纯隐式通信。现有研究主要分析 Agent 说了什么，而较少系统研究行为和环境信号如何传播。

第三，与其他 Agent 通信是绝对主流，而与人类、自身和环境同时交互的系统相对较少。

第四，逐个通信、并行通信和带总结者的并行通信都被广泛使用，没有一种策略在所有场景中占绝对优势。

这些结果说明，现有 LLM-MAS 仍然处于“人工设计消息流程”的阶段，尚未普遍实现根据任务状态动态选择通信对象、通信时机和通信内容。

Step 3：other

1：重点

这篇论文最有价值的地方，是把 LLM 多智能体系统从“多个角色组成的工作流”重新定义成了一个受通信机制约束的分布式系统。它提出的两层七维框架——架构、目标、协议、策略、范式、对象和内容——提供了一套相对完整的分析语言。对研究者而言，这意味着设计 MAS 时不应只问“需要哪些 Agent”，而应该进一步问：谁有权向谁发送什么信息，消息何时传播，如何处理冲突，错误是否会继续扩散，以及通信带来的收益是否大于成本。(arXiv)

2：意见

最关键的问题是：这套分类框架还没有成为一套可复现、可操作的科学分类方法。

作者提出了七个维度，但不少类别并不正交：例如“消息传递”和“黑板”是传输机制，“言语行为”却是消息语义；“扁平”和“层级”描述拓扑，“团队”和“社会”则描述组织关系；MCP 主要解决 Agent 与工具的连接，却与面向 Agent 间任务委托的 A2A 放在同一协议层比较。与此同时，论文没有给出系统性的检索、筛选和标注流程，也没有报告分类一致性。

3：相关工作

  1. A Survey of Agent Interoperability Protocols: MCP, ACP, A2A, and ANP

    这项工作更加专注于 Agent 协议层，系统比较 MCP、ACP、A2A 和 ANP 的交互方式、发现机制、安全模型与部署场景，并提出从工具接入到开放 Agent 网络的分阶段采用路线。与本文相比，它覆盖面更窄，但对协议细节更深入。(arXiv)

  2. EcoLANG: Efficient and Effective Agent Communication Language Induction for Social Simulation

    EcoLANG 不再只是给通信方式分类，而是直接优化“Agent 应该使用什么语言通信”。它通过压缩词汇和演化表达规则，学习一种更紧凑的 Agent 通信语言，在社会模拟中将 Token 消耗降低超过 20%，同时尽量保持模拟准确性。这是本文“通信内容与效率”方向的一个具体实现。(arXiv)

  3. Governance Gaps in Agent Interoperability Protocols: What MCP, A2A, and ACP Cannot Express

    这篇 2026 年预印本进一步指出，现有协议主要解决工具访问、能力发现、任务委托、身份和消息传递，却没有原生表达成员准入、结构化审议、投票、保留异议、人工升级和审计回放等治理语义。它实际上补充了本文较为薄弱的一层：Agent 能够通信，不等于 Agent 能够形成可靠的集体决策机制。(arXiv)
