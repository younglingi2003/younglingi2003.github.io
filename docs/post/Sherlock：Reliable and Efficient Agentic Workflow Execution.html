Sherlock：Reliable and Efficient Agentic Workflow Execution

论文解读：Sherlock

论文：Reliable and Efficient Agentic Workflow Execution
版本：我检索到的是 2025 年 11 月发布的 arXiv v1，标注为 MLSys 在审的 preliminary work。Sherlock 关注的是：工作流已经生成之后，怎样以尽可能低的成本和延迟，阻止局部错误传播到最终答案。 (arXiv)

Step 1：讲解论文

1. 论文要解决什么问题？

一个 Agentic Workflow 可以看成一张有向图：

  • 节点：一次 LLM 推理、工具调用、检索或汇总操作；
  • 边：上游节点向下游节点传递的信息；
  • 最终节点：输出用户真正需要的答案。

例如，用户要求生成一套强化学习课程：

理解需求
   ↓
生成课程大纲
  ↙       ↘
准备 Q-learning   准备 Policy Gradient
  ↘       ↙
 汇总成最终课件

这里的问题是：如果“大纲生成”节点理解错了，那么后面的所有节点都可能在错误目标上继续工作。单个节点的小错误，会沿着工作流传播甚至放大。(arXiv)

一个直接办法是给每个节点都加验证器，例如：

  • Self-Refine：让原模型自己检查并修改；
  • Advanced-Refine：让更强的大模型检查；
  • Self-Consistency：多生成几次，然后多数投票；
  • LLM-as-a-Judge：用单独的 Judge 判断；
  • Debate：让多个模型辩论后再裁决。

但论文发现，验证器本身可能让推理延迟增加最高 28.9 倍、成本增加最高 53.2 倍。如果工作流有十个节点，每个都验证，系统可能比原来贵几十倍。(arXiv)

因此，Sherlock 实际回答三个问题：

  1. 哪些节点最值得监督？
  2. 这些节点应该分配哪一种监督者？
  3. 监督过程中，怎样避免整个工作流停下来等待？

(arXiv)

2. 第一部分：找出最值得验证的节点

2.1 反事实故障注入

Sherlock 并不是简单统计“哪个节点经常答错”，而是问一个更接近因果关系的问题：

假如只让节点 (n) 出错，最终答案会变差多少？

具体过程是：

  1. 正常运行工作流，得到最终答案 (y)；
  2. 在节点 (n) 人为注入一个错误；
  3. 从节点 (n) 开始重新执行所有下游节点，得到 (y')；
  4. 比较 (y) 和 (y') 的最终正确率差异；
  5. 多次采样不同错误，求平均影响。

节点的脆弱性可以理解为：

$$ \text{Vulnerability}(n) \mathbb{E}_{\text{fault}} \big[\text{最终性能下降}\big] $$

因此，它衡量的不是节点自身有没有错，而是：

这个节点一旦出错，会对最终结果造成多大影响。

论文注入三类故障：

故障类型 模拟方式 含义
行为偏离 替换或修改节点提示词 Agent 误解了自己的角色或任务
上下文丢失 删除部分上游信息 Agent 没收到完整历史信息
执行错误 将节点输出替换为错误结果 推理、工具调用或格式发生错误

三类错误频率参考了真实多智能体执行日志中的错误分布，其中执行错误约占 52.69%，行为偏离约占 28.63%，上下文丢失约占 18.68%。(arXiv)

2.2 得到的结构性规律

作者在 100 多张工作流图、超过 1.5 万条执行轨迹上进行故障注入，得到三个主要规律：

第一，最终节点最脆弱。

最终节点后面没有其他 Agent 帮它纠错。它一旦汇总错误，错误会直接进入最终答案。

第二，初始节点也比较重要。

最开始对任务理解错误，会使整个工作流沿着错误目标执行。

第三，入度越高的节点越危险。

入度高意味着这个节点要汇总多个上游结果，它既可能继承多个上游错误，也可能在汇总过程中放大冲突。相反，论文没有观察到出度，即下游分支数量，与节点脆弱性存在明显相关性。(arXiv)

于是论文给出一个很简单的验证优先级：

最终节点
  >
初始节点
  >
中间节点，按入度从高到低排序

如果用户只允许验证两个节点，就选前两个；如果预算更多，再逐步加入高入度的中间节点。(arXiv)

我的看法

这里最有价值是论文提出了一个明确的研究范式：

通过局部反事实干预，测量局部错误对全局结果的影响。

3. 第二部分：动态选择最合适的验证器

确定节点需要验证之后，下一个问题是：

应该给它分配 Self-Refine、Debate、Judge，还是更大的模型？

Sherlock 发现，不存在一个验证器在所有任务上都最好。

例如，实验中：

  • 代码任务的基线准确率约为 0.51，Self-Refine 反而降到约 0.31；
  • 数学任务中，Debate 从基线约 0.62 降到约 0.45；
  • 工具调用任务中，Self-Consistency、Judge 和 Debate 效果较好；
  • Advanced-Refine 使用 70B 模型，但高成本并不总能带来最高准确率。

因此，Sherlock 为每个验证器定义效用：

$$ U(v_i,\tau)=P(v_i,\tau)-\lambda C(v_i,\tau) $$

其中：

  • $(P(v_i,\tau))$：验证器对任务 $(\tau)$ 带来的准确率提升；
  • $(C(v_i,\tau))$：额外计算成本；
  • $(\lambda)$：用户有多在意成本。

当 $(\lambda)$ 小，系统更愿意使用昂贵但强大的验证器；当 $(\lambda)$ 大，系统更倾向于选择便宜的验证方法。(arXiv)

在实现上，论文使用 DistilBERT 编码任务提示词，再通过线性层预测每个候选验证器的偏好分数。训练时，用同一任务下不同验证器之间的相对效用作为 advantage，采用作者所称的 GRPO 式组内相对偏好优化。(arXiv)

通俗地说，它学习的是：

输入：这个节点当前负责什么任务？
输出：
Self-Refine       0.10
Debate            0.15
LLM-as-a-Judge    0.50
Advanced-Refine   0.05
Self-Consistency  0.20

然后选择分数最高的验证器。

我的看法

Sherlock 已经包含：

  • 一组功能不同的验证器；
  • 根据任务动态选择验证器；
  • 同时考虑准确率和成本。

4. 第三部分：推测执行与回滚

即使只验证少量节点，验证过程仍然可能很慢。

传统执行方式是：

执行 W1
等待验证 W1
执行 W2
等待验证 W2
执行 W3

Sherlock 改成：

执行 W1
├── 后台验证 W1
└── 同时继续执行 W2、W3

如果 W1 验证通过，W2、W3 的结果直接保留；如果 W1 被验证器修改，依赖 W1 的下游结果可能失效，于是系统回滚并重新执行。(arXiv)

这类似 CPU 的 speculative execution：

先假设前面的结果没问题，继续向前执行；如果后来发现猜错，再撤销。

Sherlock 对推测深度设置两个限制：

  1. 下游节点的累计执行时间不能超过当前验证器的执行时间；
  2. 推测执行产生的预期浪费成本不能超过用户预算。

推测越深，延迟越低；但一旦回滚，浪费的计算越多。(arXiv)

选择性回滚

验证器修改原输出，并不意味着下游结果一定失效。

Sherlock 使用轻量相似度指标判断是否回滚。实验发现：

  • 在指令和工具任务上，ROUGE-L 与语义等价标签相关性较好，AUC 分别约为 0.828 和 0.877；
  • 在代码和数学任务上，所有轻量指标基本接近随机判断，甚至出现负相关。

因此，对指令和工具任务，Sherlock 使用 ROUGE-L 阈值决定是否回滚；对代码和数学任务，则保守地全部回滚。(arXiv)

我的看法

这一部分是论文中最具有“系统论文味道”的设计。它没有单纯降低验证次数，而是通过并行执行隐藏验证延迟。

不过，它依赖一个重要前提：工作流节点必须具有可回滚性。对于发送邮件、支付、删除文件、修改数据库等带有外部副作用的工具调用，简单“撤销下游结果”并不成立。

5. 一句话总结核心方法

Sherlock 的本质是：

先用错误传播分析决定“哪里需要监督”，再用学习模型决定“由谁监督”，最后用推测执行解决“监督太慢”的问题。

它同时优化三个目标：

$$ \text{准确率}\uparrow,\qquad \text{成本}\downarrow,\qquad \text{延迟}\downarrow $$

其真正的新意不在某个单独模块，而在于把错误传播、验证器路由和运行时调度整合成一个完整系统。(arXiv)

Step 2：实验部分

1. 实验设置

论文使用一台配有 8 张 A100 80GB 的服务器：

功能 模型
主执行 Agent Llama-3.1-8B-Instruct
第二执行模型 Qwen2.5-7B-Instruct
高级反馈模型 Llama-3.3-70B-Instruct
Judge 模型 Selene-1-Mini-Llama-3.1-8B

工作流由 Flow Planner 动态生成。主要测试三个基准：

  • CoTCollection：多步指令理解和推理；
  • OMEGA compositional subset：组合数学推理；
  • LiveCodeBench：代码生成、执行和输出预测。

成本、准确率和延迟分别单独测量。(arXiv)

2. 实验一：验证器真的需要动态选择吗？

答案是需要。

不同验证器在不同任务上的表现差异很大，而且成本和准确率并非单调关系。某些验证器不仅没有提升结果，反而因为错误反馈或冲突判断降低准确率。

我的判断：这个实验很重要，因为它直接证明了“动态监督者路由”不是纯粹为了节省成本，而是为了避免错误监督。监督者并不天然可靠，选择错误的监督者可能比不监督更差。

3. 实验二：节点放置策略是否有效？

作者在相同验证预算下比较：

  • 随机选择节点；
  • 均匀间隔选择节点；
  • Sherlock 的拓扑策略。

结果显示，在大多数预算水平下，Sherlock 用更少的验证节点达到更高准确率，尤其是在预算很小时优势明显。其原因是优先覆盖了最终节点、初始节点和高入度聚合节点。

但随着验证预算接近覆盖所有节点，不同策略的差距逐渐缩小。这说明 Sherlock 主要解决的是：

预算有限时，验证资源应该优先投向哪里。

4. 实验三：验证器选择是否有效？

作者比较了：

  • 所有任务固定使用同一个验证器；
  • 按“数学、代码、指令、工具”类别查表；
  • AFlow 的搜索式方法；
  • Sherlock 的动态选择器；
  • 知道正确答案的 Oracle。

结果显示，Sherlock 的准确率—成本点基本落在 Pareto frontier 上，即在相同成本下获得更高准确率，或者在相同准确率下花费更少成本，并接近 Oracle 的效果。与不验证基线相比，论文报告平均准确率提高 18.3%；与基于 Monte Carlo 搜索的方法相比，验证成本降低约 26.0%。(arXiv)

我的判断：结果支持动态选择器有效，但这里的 Oracle 是“事后知道哪个验证器能给出正确答案”的理想上界，现实系统无法获得。另外，Sherlock 的离线训练数据需要在训练任务上运行全部候选验证器，域接入成本并不低，只是被移到了离线阶段。

5. 实验四：推测执行能降低多少延迟？

平均延迟下降如下：

数据集 执行完成时间 $(T_{\text{exec}})$ 最终验证完成时间 $(T_{\text{vrf}})$
CoTCollection 51.4% 21.9%
OMEGA 53.4% 30.6%
LiveCodeBench 62.9% 48.7%

LiveCodeBench 提升最大，因为代码工作流中的节点执行和验证时间都较长，能够与更多下游计算重叠。(arXiv)

但有一个值得注意的细节：OMEGA 的 P99 延迟下降只有约 4.8% 和 5.5%。这意味着平均延迟改善明显，但在部分极端工作流上，回滚、图结构或验证器时延可能使尾延迟收益接近消失。(arXiv)

实验部分总体评价

实验的优点是组件拆分比较清楚：

  • 节点放置只和随机、均匀放置比较；
  • 验证器选择单独比较成本—准确率；
  • 推测执行单独测量延迟。

问题是评测仍然围绕一个 Flow Planner、三类基准和一组固定模型展开。它证明了系统在该实验配置下有效，但尚未充分证明拓扑规律和验证器选择器可以跨 Planner、跨模型和跨真实业务工作流稳定泛化。

Step 3：意见、最终判断

意见

论文对“节点脆弱性”的建模过度依赖合成故障和静态拓扑规律，尚不足以支撑其跨工作流泛化主张。

作者通过提示词替换、上下文删除和输出替换模拟错误，再归纳出“终点优先、起点其次、高入度中间节点优先”的规则。但真实 Agent 错误往往具有以下特点：

  • 错误与任务内容、模型能力和工具类型相关；
  • 不同节点错误可能相关，而不是一次只错一个节点；
  • 下游 Agent 有时能纠错，有时会盲目服从；
  • 同一拓扑位置在不同语义工作流中的重要性可能完全不同；
  • 固定温度为 0 的故障实验不能覆盖真实生产环境中的采样波动。

论文至少需要补充跨 Planner、跨模型、跨领域的 OOD 实验，并使用真实失败轨迹验证：合成故障得到的节点排序，是否真的与线上错误传播风险一致。现有结果更像是对有限实验分布有效的经验规则，而不是可靠的通用脆弱性估计器。相关结论目前建立在 100 多张图和约 1.5 万条注入轨迹上。(arXiv)

最终判断

Weak Accept，6/10。

理由是：问题重要，系统设计完整，错误传播分析、验证器路由和推测执行的组合具有明确价值，实验也展示了不错的准确率—成本—延迟权衡。但其最核心的节点脆弱性策略仍较启发式，跨域和真实生产环境的外部有效性不足。作为 MLSys 类系统论文可以弱接收；若按强调方法普适性的顶级机器学习会议标准，则更接近 Borderline。

Step 4：简要汇报

Sherlock面向多节点Agent工作流中的错误传播和验证开销问题。它先通过反事实故障注入分析节点对最终结果的影响，优先验证终点、起点和高入度节点；随后根据任务动态选择Self-Refine、Debate、LLM-as-a-Judge等成本最优验证器；运行时让验证与下游节点并行，并在发现错误后选择性回滚。实验平均提升18.3%准确率，并显著降低验证成本与执行延迟。其不足是节点风险仍主要依赖合成故障和静态拓扑规律，跨域泛化尚未充分验证。(arXiv)

Step 5：相关工作

1. 输出验证与推理增强

Self-Refine 让同一个模型依次生成答案、提供反馈并修改答案；Self-Consistency 通过采样多条推理路径并多数投票选答案；Multi-Agent Debate 让多个模型多轮讨论；LLM-as-a-Judge 使用强模型评估候选答案。Sherlock 没有重新发明这些验证器，而是把它们作为候选监督操作，研究何时、何地以及选择哪一种。(arXiv)

2. 自动工作流生成与优化

ADAS 使用 Meta Agent 自动编写和搜索 Agent 系统；AFlow 把工作流表示成代码图，并通过 Monte Carlo Tree Search 优化；Flow 使用活动节点图动态调整子任务和工作流结构。这些工作主要回答“工作流应该长什么样”，Sherlock 回答“工作流生成以后，怎样可靠、低成本地执行”。(arXiv)

3. 节点模型与组件路由

LLMSelector 研究复合 AI 系统中每个模块应该使用哪个模型，并通过模块级性能估计减少组合搜索。它和 Sherlock 的验证器选择非常接近，但 LLMSelector 路由的是执行模型，Sherlock 路由的是 Self-Refine、Debate、Judge 等验证机制。(arXiv)

4. Agent 工作流服务系统

Murakkab 从工作流、模型和硬件三个层面联合优化资源配置；Circinus 根据延迟、成本和准确率 SLO 规划复合 AI Pipeline；Autellix 把整个 Agent 程序而非单次 LLM 请求作为调度单位。它们更侧重资源调度和系统吞吐，而 Sherlock 的重点是验证器放置、错误恢复和可靠性。(arXiv)

5. Agent 推测执行

Speculative Actions 使用快速模型预测 Agent 接下来可能执行的动作，并提前并行执行。Sherlock 的推测对象不同：它不预测下一步，而是在当前节点验证尚未完成时，假设原输出正确并继续运行已知的下游工作流；验证失败后再回滚。(arXiv)
