BrowseComp-Plus

论文解读：BrowseComp-Plus: A Fair and Disentangled Evaluation Benchmark for Deep Search Agents

论文最初以技术报告形式发布于 2025 年 8 月，目前已经以 ACL 2026 Main Conference 长论文正式发表，最终标题改为 BrowseComp-Plus: A Fair and Disentangled Evaluation Benchmark for Deep Search Agents。下面以你给出的 arXiv 版本为主，同时参考正式版本中补充的实验与局限性讨论。(arXiv)

Step 1：论文讲解

1. 这篇论文解决什么问题？

这篇论文提出一个用于评测 Deep Research Agent 的新基准。

Deep Research Agent 一般执行下面的循环：

分析问题 → 生成搜索词 → 调用搜索引擎 → 阅读结果 → 判断缺少什么 → 继续搜索 → 汇总证据 → 给出答案

此前最有代表性的评测是 OpenAI 的 BrowseComp，其中包含 1266 个非常难搜索、但答案通常很短的问题。例如，问题不会直接问“某个人是谁”，而是同时给出职业、经历、时间、作品等多个线索，要求 Agent 在互联网中把这些线索拼起来。(OpenAI)

问题在于，BrowseComp 使用的是实时互联网和商业搜索 API。因此最终成绩实际上混合了很多因素：

$$ \text{最终成绩}= \text{LLM能力} + \text{搜索引擎能力} + \text{当时互联网状态} + \text{工具接口设计} $$

例如，两个研究团队都测试同一个模型：

  • 一组使用 Google Search；
  • 一组使用 Bing；
  • 一组在 2025 年测试；
  • 一组在 2026 年测试；
  • 搜索引擎可能已经偷偷更新排序算法；
  • 某些网页可能消失或出现；
  • 测试答案甚至可能已经泄露到互联网。

这时模型分数不同，并不代表 Agent 本身更强。

BrowseComp-Plus 的核心思想就是：把互联网搜索实验搬进一个固定的“实验室”。

它固定：

  • 问题；
  • 文档集合；
  • 正确证据文档；
  • 干扰文档；
  • 搜索接口；
  • 检索器；
  • 评价指标。

这样便可以分别研究：

  1. Agent 会不会规划和搜索；
  2. Retriever 能不能找到正确文档；
  3. Agent 找到文档后能不能正确推理；
  4. Agent 是否引用了真正支持答案的证据。

这就是论文所说的 disentangled evaluation，即解耦评测。(ar5iv)

2. BrowseComp-Plus 是怎么构造出来的？

第一阶段：为问题寻找完整证据

原始 BrowseComp 只有：

  • 问题；
  • 标准答案。

但没有告诉研究者：

  • 哪些网页能证明答案；
  • 每个线索分别由哪个网页支持；
  • 推理过程中需要哪些文档。

于是作者把“问题 + 标准答案”交给带网页搜索能力的 OpenAI o3，让它反向查找证据。

o3 的输出被组织成：

字段 含义
Clue 问题中的一个线索
URL 支持该线索的网页
Evidence 网页中具体支持该线索的文字

在原始 1266 个问题中：

  • o3 对 124 个问题没有成功提供证据；
  • 另有 137 个问题因为网页无法抓取等原因被排除；
  • 最终有 1005 个问题进入人工验证。(ar5iv)

第二阶段：人工验证证据链

14 名大学生标注员对每个问题进行检查。

他们不是简单判断“这个网页看起来相关”，而是要完成两个任务：

  1. 在网页中标出具体支持某个线索的文本；
  2. 确认所有文档组合起来，确实能够完整推出答案。

如果 o3 找到的文档不足，标注员需要继续搜索至少 20 分钟。

最终：

  • 830 个问题通过人工验证；
  • 总标注时间超过 400 小时；
  • 抽样交叉验证的一致率平均超过 80%。(ar5iv)

这里有两种重要文档：

Evidence document

能够支持问题中某一个或多个线索的文档。

Gold document

能够直接或隐式包含最终答案的文档。

例如问题问：“某作者共发表了多少篇论文？”

作者主页可能没有写“7篇”，但列出了七篇论文。它仍然属于 gold document，因为答案可以直接从页面推出。

这种定义比简单搜索答案字符串更加合理。(ar5iv)

第三阶段：加入困难负样本文档

如果数据集中只有正确文档，检索器很容易找到答案。

因此作者使用 GPT-4o 将每个复杂问题分解为平均大约 7 个子查询，然后把子查询交给 Google Search，经由 SerpAPI 获取最多 100 条结果。

这些结果通常和问题高度相关，但并不能完整支持答案，因此构成 hard negatives。

例如问题真正寻找的是一篇特定论文，但负样本文档可能是：

  • 同主题的其他论文；
  • 同名作者；
  • 时间接近但作者不符合的论文；
  • 只符合部分线索的网页。

最终得到：

  • 830 个问题
  • 100,195 篇文档
  • 每题平均 6.1 篇证据文档
  • 每题平均 2.9 篇 gold 文档
  • 每题平均 76.28 篇专门挖掘的困难负样本
  • 每篇文档平均约 5179 个单词。(ar5iv)

3. Agent 在这个基准上怎么工作？

Agent 看不到答案和证据标签，只能看到问题，并获得一个搜索工具。

每次调用搜索工具：

  1. Agent 生成搜索查询；
  2. Retriever 在固定的 10 万篇文档中检索；
  3. 返回前 5 篇文档；
  4. 默认只展示每篇文档前 512 tokens；
  5. Agent 判断是否继续搜索；
  6. 最后输出答案、置信度和引用文档。

作者测试了不同组合：

Agent 模型

  • GPT-4.1
  • o3
  • GPT-5
  • Claude Sonnet 4
  • Claude Opus 4
  • Gemini 2.5 Pro / Flash
  • gpt-oss-120B
  • Qwen3-32B
  • Search-R1-32B

Retriever

  • BM25：传统关键词匹配；
  • Qwen3-Embedding：0.6B、4B、8B；
  • ReasonIR：面向推理检索训练的模型；
  • 正式版本还加入 Jina-ColBERT-v2。

这样可以固定 Retriever 比较 Agent，也可以固定 Agent 比较 Retriever。

4. 这篇论文最重要的结论

结论一：Retriever 对 Agent 的影响非常大

很多人会认为，大模型足够强之后，检索器差一点也没关系。

这篇论文给出的结论恰好相反。

例如：

Agent BM25 Qwen3-Embedding-8B
GPT-4.1 14.58% 35.42%
o3 49.28% 63.49%
GPT-5 55.90% 70.12%
Sonnet 4 14.34% 36.75%
Opus 4 15.54% 36.14%
gpt-oss-120B-high 28.67% 42.89%

换一个 Retriever，GPT-4.1、Sonnet 4、Opus 4 的正确率甚至翻倍。

更强的 Retriever 通常还会减少搜索次数。例如 GPT-5 从平均 23.23 次下降到 21.74 次。这说明好的检索器不仅让答案更准确，还能让 Agent 更早找到有效证据，不必不断试错搜索。

我的看法： 这是全文最有价值的实证结论。当前 Agent 研究经常把精力放在更复杂的规划、反思和 RL 上，但很多失败可能只是第一轮检索结果太差。一个更好的 Retriever，可能比增加多轮反思更便宜、更稳定。

结论二：开源模型的主要问题不一定是“不会推理”，而是“不会持续使用搜索工具”

在 Qwen3-Embedding-8B 下：

  • GPT-5：70.12%
  • o3：63.49%
  • gpt-oss-120B-high：42.89%
  • GPT-4.1：35.42%
  • Qwen3-32B：10.36%
  • Search-R1-32B：10.36%

关键区别还体现在搜索次数：

  • GPT-5 和 o3 平均搜索 20 多次；
  • Qwen3-32B 平均搜索不到 1 次；
  • Search-R1 平均搜索约 1.7 次。

也就是说，很多开源模型没有真正进行“Deep Research”，而是搜索一两次就急着回答。

这也解释了为什么专门训练过搜索的 Search-R1 仍然只有约 10%：

  • 它可能学会了调用搜索工具；
  • 但没有学会根据搜索结果动态修改假设；
  • 没有学会发现缺失证据；
  • 没有学会持续搜索直到所有线索闭合。

换句话说：

会调用一次 search，不等于会进行多轮研究。

结论三：当前 Retriever 本身离“解决复杂检索”还很远

作者直接用完整问题作为查询，测试 Retriever 能否找到证据。

以 evidence documents 为标准：

Retriever Recall@5 Recall@100 Recall@1000
BM25 1.2% 4.7% 13.7%
Qwen3-Embedding-0.6B 6.2% 26.5% 59.7%
Qwen3-Embedding-4B 9.8% 40.2% 71.8%
Qwen3-Embedding-8B 14.5% 47.7% 76.7%
ReasonIR-8B 12.2% 43.6% 73.9%

即使最好的 Qwen3-Embedding-8B，在前五篇结果中也只能召回约 14.5% 的证据。

原因是复杂问题往往包含多个线索，将复杂问题一次性编码成一个 embedding，未必能很好匹配每一个局部证据。

这说明复杂搜索不应只是：

把原问题送进 Retriever。

而应该是：

识别线索 → 分解子问题 → 分别检索 → 根据证据生成下一轮查询。

这也是 Agentic Retrieval 真正值得研究的地方。(ar5iv)

Step 2：实验部分

实验一：Oracle Retrieval

作者做了一个非常重要的“上限实验”：

不让 Agent 自己搜索，直接把所有正确证据文档交给它。

结果：

  • GPT-4.1：93.49%
  • Qwen3-32B：83.25%

对比正常检索时：

  • GPT-4.1 + BM25：14.58%
  • GPT-4.1 + Qwen3-Embedding：35.42%
  • Qwen3-32B + Qwen3-Embedding：10.36%

这个结果说明：

大量错误并不是因为模型看完证据后不会回答，而是因为模型根本没有把证据找全。

而且当正确文档全部给出后，GPT-4.1 和 Qwen3-32B 的差距明显缩小。

因此开源模型与闭源模型之间的巨大差距，至少部分来自：

  • 搜索意愿；
  • 查询生成；
  • 证据缺口识别；
  • 多轮工具调用；
  • 检索结果利用。

而不完全来自模型底层知识或静态推理能力。(ar5iv)

我的看法： 这是论文里最能指导后续研究的实验。它告诉我们，不要看到 Agent 答错，就立刻认为要强化 reasoning。更合理的诊断顺序应该是：

  1. 正确证据是否进入上下文？
  2. 证据是否被 Agent 识别为有用？
  3. 证据齐全时模型是否能推出答案？
  4. 最后才考虑是否需要增强推理能力。

实验二：增加 reasoning effort

作者用 gpt-oss 的 low、medium、high 三种推理强度研究测试时计算量。

以 gpt-oss-20B + Qwen3-Embedding-8B 为例：

推理强度 Accuracy Recall 搜索次数
Low 13.37% 17.37% 1.87
Medium 29.88% 41.31% 13.64
High 34.58% 49.29% 23.87

从低推理强度到高推理强度：

  • 正确率大幅提升；
  • 证据召回明显提升；
  • 搜索次数从约 2 次增加到约 24 次。

这说明 Deep Research 中所谓的 test-time scaling，不只是模型在脑内“想得更久”，还表现为：

  • 更愿意尝试不同查询；
  • 更频繁检查已有证据；
  • 搜索失败后继续尝试；
  • 更有可能找到完整证据链。

但代价是调用量、延迟和 token 成本迅速增加。

我的判断是，这个实验支持 test-time scaling，但没有证明“搜索越多越好”。更准确的说法是：

更强的模型往往既搜索得更多，也搜索得更有效。

搜索 20 次的弱策略可能只是重复试错。真正值得研究的是，在固定预算下如何选择信息增益最大的下一次搜索。

实验三：完整文档读取工具

主要实验只向 Agent 展示每篇文档前 512 tokens。作者随后增加 get-document 工具，允许 Agent读取完整文档。

结果：

模型 默认设置 加入全文读取
GPT-4.1 35.42% 43.61%
Qwen3-32B 10.36% 11.69%

GPT-4.1 每题平均调用全文读取工具 1.85 次，而 Qwen3-32B 仅调用 0.27 次。(ar5iv)

这说明工具本身存在，并不等于模型会正确使用它。

一个好的 Agent 需要判断：

  • 当前摘要是否已经足够；
  • 关键信息是否可能位于文档后半部分；
  • 哪一篇文档值得阅读全文；
  • 阅读全文后是否需要更新搜索计划。

我的看法： 这个实验实际上证明了 context engineering 与 Agent policy 是耦合的。不是简单地“给更多上下文就会更好”，而是模型必须学会何时请求更多上下文。

实验四：成本和引用

更强的 Retriever 不仅提升准确率，还降低部分闭源 Agent 的 API 成本。例如：

  • GPT-5：约 400.36 美元降到 360.71 美元；
  • o3：约 836.35 美元降到 740.79 美元；
  • GPT-4.1：约 106.96 美元降到 89.81 美元。

这是因为有效结果出现得更早，Agent 需要的搜索和上下文 token 更少。(arXiv)

正式版本还发现，Agent 即使检索到了证据，也不一定会引用它。Citation Recall 通常低于检索 Recall，说明 Agent 存在第二层失败：

找到了正确文档，但没有意识到它应该成为最终答案的证据。

Step 3：意见

总体评价

这是一篇基础设施型、benchmark 型论文。算法创新不强，但问题重要，数据构建投入较大，并且实验能够得出对 Agent 研究很有价值的诊断性结论。

优点

1. 问题抓得非常准

论文准确指出，实时网页搜索评测把 Agent、Retriever、搜索引擎和网页变化混在了一起。

固定 corpus 和 Retriever 后，研究者终于可以回答：

  • 是 Agent 不会搜索，还是搜索器太差？
  • 是没有找到证据，还是找到后不会推理？
  • 增加搜索次数是否有效？
  • 替换 Retriever 能带来多少收益？

这比单纯公布一个最终准确率更有科学价值。

2. 数据构建相对扎实

作者不是仅用模型自动生成证据，而是要求人工逐线索验证、标记证据片段，并检查整个证据链是否充分。

830 个问题、10 万篇文档、400 多小时人工工作，使其具备成为共享评测平台的基础。数据、代码和复现实验流程也已经公开。(ar5iv)

3. 实验具有明确的诊断意义

特别是三个结论很有价值：

  • 好 Retriever 可以显著提升 Agent，并降低搜索成本；
  • oracle evidence 下开源与闭源模型差距显著缩小；
  • 开源 Agent 的明显短板是持续、多轮、交错式搜索，而不只是最终推理。

这些结果能够直接影响后续 Agent 训练和系统设计。

最终判断

这篇论文最核心的贡献可以概括为：

它把 Deep Research Agent 的评测，从不可控的实时网页考试，变成了可以拆解、复现和诊断的实验。

它最重要的发现不是“GPT-5 得分最高”，而是：

  1. Retriever 仍然是 Deep Research 系统的核心瓶颈；
  2. 很多开源 Agent 的主要短板是持续搜索和证据闭环，而不只是静态推理；
  3. 只要正确证据进入上下文，现有模型已经能解决大多数问题；
  4. 更好的搜索并不一定意味着更多搜索，而是更早找到高价值证据。
